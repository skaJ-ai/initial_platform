## A. 아키텍처 정합성

### A0. 왜 HR AX는 별도 Platform인가?

**A** (확정): 원문 HR Data와 HR Knowledge는 전사 공통 AI Platform에 올리지 않는다.

전사 공통 AI Platform과 사외 LLM은 모델·기능으로 연계할 수 있지만, HR 데이터의 보안 등급, 조직×도메인 권한, 검수 책임, 자산 공개 범위는 HR AX가 통제해야 한다. 그래서 HR AX는 단순 챗 UI나 공통 Platform 위 플러그인이 아니라 **AI Tool 앞단에서 HR Data/Knowledge를 별도 적재·통제하는 안전 게이트 Platform**이다.

핵심 이유는 세 가지다.
- **보안**: AI Tool 앞단의 안전 게이트. 기밀/민감 HR 데이터와 Knowledge의 원문 적재 금지, 분류·차폐·audit
- **축적**: Light user가 직접 RAG/DB를 관리하지 않아도 Work Card 사용 흐름에서 자산화
- **권한**: 조직×도메인 권한 단위별 DW·표준 데이터·자산 데이터 접근 통제

### A1. Agent / Skill / Tool은 어디에 사는가?

**A** (확정): 모두 플랫폼 안에 위치. 사외 LLM 호출 시 호출별 컨텍스트로 주입.

#### Agent — 두 종으로 분리

**Domain Agent (도메인 특화)**
- 특정 도메인 내 특정 업무에 특화된 LLM 처리 단위
- 예: 서류평가 Agent (채용/L3), 인사위 안건 정리 Agent (임원조직/L5), 입과 안내 Agent (교육/L6)
- Skill 묶음 + Tool 권한 + 프롬프트 흐름으로 구성

**System Agent (플랫폼 운영)**
- 사용자에게 보이지 않는 플랫폼 운영 단위
- 예: **데이터 민감도 분류 Agent** (기밀/민감/일반 분류), 차폐 검사 Agent, 자산 추출 Agent
- 호출 모델: 사내 LLM 단독 (사외 LLM 절대 사용 X)

#### Skill
특정 업무를 진행하기 위한 노하우. md 형식. LLM에 컨텍스트로 주입.
예: `서류평가-특이사항-하이라이트-기준.md`, `임원-평가-코멘트-익명화-기준.md`

#### Tool
결정적 코드 실행 단위. Function Calling 패턴.
예: 임직원 정보 조회, 잔여휴가 계산기, 출장비 계산기

#### 보관 위치 (M11 결정)
- 정의(코드): 플랫폼 코드 리포지토리 (Lab 관리, git)
- 메타·임베딩: 별도 DB
- 사용: Workbench에서 카드 진입 시 시스템 자동 매칭 부착

### A2. 사외 LLM과 Tool/Skill/Agent의 관계?

**A** (확정): 사외 LLM은 직접 접근하지 않음. **Function Calling** 패턴.

```
1. 플랫폼이 사외 LLM에게 사용 가능한 Tool 목록 + Skill 컨텍스트 묘사 전달
   (Tool 코드 자체는 보내지 않음. 이름·인자·설명만)
2. 사외 LLM이 "Tool X를 인자 Y로 호출하겠다"는 의도 반환
3. 플랫폼이 Tool 실제 실행 (사내에서)
4. 결과를 비식별화 게이트 통과 후 사외 LLM에 컨텍스트로 전달
5. 사외 LLM이 최종 응답 생성 → 사내 재매핑 → 사용자
```

**한 줄**: 사외 LLM은 의도만 표현. 실제 실행·데이터 노출 통제는 플랫폼이 전담.

### A2-1. 여기서 말하는 AI Tool은 무엇인가?

**A**: 다음 3종으로 구분한다.

| 구분 | 예 | 원칙 |
|---|---|---|
| 사내 LLM | GAUSS | LLM 자체 로그 미저장. 원문 HR 데이터 처리 가능. HR AX application-side audit만 기록 |
| 사외 LLM | Chat GPT / Gemini 등 | 원문 HR Data/Knowledge 직접 접근 금지 |
| 사내 AI Tool | 전사 AX팀이 제공하는 일부 AI Tool | HR AX의 권한·분류·차폐 정책 통과 후 제한 연계 |

결재, 메일, 캘린더, HR 시스템 API는 AI Tool이 아니라 일반 업무 Tool 또는 원천 시스템 연계다. 이들도 필요하면 HR AX 내부 Tool로 실행될 수 있지만, AI Tool 연계 범위에 넣어 설명하지 않는다.

※ 사외 LLM은 원문 HR Data/Knowledge를 직접 보지 않는다. HR AX가 권한 확인, 비식별화, 감사 로그를 거친 차폐 컨텍스트만 전달한다.

### A3. Skill md는 누가 작성하고 어디에 저장되는가?

**A** (확정):
- **초안**: Domain Experts가 자동 추출 후보를 검토해 md 형식으로 작성
- **공식 등록**: AX&CI Lab이 Skill md를 정형화·등록
- **승격 경로 (자산 → Skill)**: 자동 추출 (System Agent 보조) → Domain Experts 초안 → AX&CI Lab 정형화·등록
- **저장**: 플랫폼 코드 리포지토리(정의) + 별도 DB(메타·임베딩)
- **데이터 종에 속하지 않음**: 실행 레이어 자료. ② 표준 데이터와 별개

### A4. L4/L5/L6 프로세스 트리는 어느 데이터 종에 속하는가?

**A** (확정): ② 표준 데이터(Standards)에 위치. AX&CI Lab이 운영.

### A5. ② 표준 데이터의 정확한 범위는?

**A** (확정): **규정 / 기준 / 업무 프로세스 한정**.
- 인사 제도, 인사 기준 (근태·평가·보상), 업무 프로세스 트리(L4/L5/L6), 표준 양식
- Skill md는 ② 안에 들어가지 않음 (실행 레이어 자료)

### A6. ③ → ② 자산 → 표준 승격 조건?

**A** (확정): 자산이 "규정/기준/업무 프로세스" 형태로 정형화 가능할 때만.
- 주체: AX&CI Lab + 부문 People팀 합의
- 결과: 새 L6 노드 또는 기준 항목 추가
- 빈도: 드뭄
- 그 외 노하우는 ③ 자산 데이터에 영원히 잔류

### A7. ③ → Skill 승격 조건?

**A** (확정): 자산이 LLM 운영 노하우로 추출 가능할 때.
- 주체: AX&CI Lab (Domain Experts 초안 받음)
- 결과: 새 Skill md 등록
- 빈도: ② 승격보다 잦음

### A8. 자동 포착 후 L6 vs Skill 어떻게 구별?

**A** (확정): 자동 포착은 동일 메커니즘. **Lab이 검토 시 4분기 결정**.
- 새 L6 (프로세스 단위)
- 새 Skill (노하우 md)
- 둘 다
- 둘 다 아님 (반려)

## B. 데이터 등급과 처리

### B1. 데이터는 몇 단계로 분류되는가?

**A** (확정): **3단계** — 기밀 / 민감 / 일반.

| 등급 | 처리 | 모델 |
|---|---|---|
| 기밀 | 사외 송출 금지 | 사내 LLM(GAUSS, 로그 미저장) |
| 민감 | 비식별화 게이트 후 외부 | 사외 LLM(Chat GPT / Gemini 등) |
| 일반 | 그대로 외부 | 사외 LLM |

### B2. 자동 분류는 어떻게 되는가?

**A** (확정): **데이터 민감도 분류 Agent**가 통합 처리.

```
데이터 민감도 분류 Agent (System Agent, 사내 LLM 전담)
 ├─ 1차: 조직×도메인 권한 단위 기본 등급 룩업 (분류표 v0)
 ├─ 2차: PII 패턴 매칭 (사번/이름/평가/징계 등)
 ├─ 3차: 사내 LLM(GAUSS) 한국어 맥락 보강
 └─ 출력: {등급, 차폐 후보 항목, 신뢰도}

사용자 상향 단계:
 - 분류 결과를 사용자에게 노티
 - 사용자는 상향만 가능 (하향 권한 없음)
 - 상향 시: 즉시 적용 + audit log + 권한 단위 Domain Expert 통지
 - N회 이상 상향 시 분류표 v0 갱신 트리거
```

### B3. GAUSS는 어떤 역할인가?

**A** (확정):
- ⭕ 기밀 데이터 응답 생성 (사내 처리)
- ⭕ 자산 사내 임베딩 (retrieval용)
- ⭕ PII detection 보조 (선택)
- ❌ 민감/일반 응답 생성에는 사용 안 함

## C. 자산화 운영

### C1. "셀"은 무엇인가?

**A**: 이전 문서에서 쓴 "셀"은 **조직×도메인 권한 단위**를 줄여 부른 말이다. 대외 문서에서는 "셀"보다 "조직×도메인 권한 단위"를 쓴다.

예: `부문 People팀 × 보상`, `MX사업부 × 채용`, `VD사업부 × 임원조직`.

이 단위가 DW 접근 범위, 표준 데이터 RAG 범위, 자산 데이터 공개 범위, 차폐 정책, Domain Experts를 결정한다.

### C2. Domain Experts는 누구인가?

**A**: 부문 또는 사업부에서 선별된 담당자. 조직×도메인 권한 단위별 검수 책임.

### C2-1. 권한 단위 담당자가 1인일 경우 검수는?

**A** (확정): **본인 자가 검수 가능**.
- 자가 검수 자산은 디폴트 "권한 단위 한정" 또는 "개인" 만 허용
- 도메인 종방향 승격 시 부문 Domain Expert 추가 검수 필수
- 후임 변경 시 권한 단위 한정 자산은 자동 retrieval

### C2-2. 권한 단위 담당자가 2인 이상일 경우?

**A** (확정): 같은 권한 단위 담당자 간 교차 검수 필수. 본인 자산을 본인이 검수 X.

### C2-3. 자산 공개 범위 결정 주체?

**A** (확정): **작성자 제안 + 검수자 확정 + 단계적 권한 escalation**.

```
1. 작성자가 카드 종료 시 공개 범위 제안
   - 시스템 디폴트 추천 (대부분 "권한 단위 한정")
2. 검수자가 승인 시 범위 확정
   - 좁히는 것은 단독 가능 (예: 권한 단위 → 개인)
   - 넓히는 것은 단계적:
     · 권한 단위 → 도메인 종방향: 부문 Domain Expert 추가 검수
     · 도메인 종방향 → 전사 표준: AX&CI Lab 추가 검수
```

원칙: 디폴트는 좁게, 좁히는 건 단독 OK, 넓히는 건 권한 escalation.

### C3. Domain Experts 인센티브는?

**A**:
- 승인 자산은 Domain Expert와 작성자 양쪽 실적
- 검수 행위 자체도 실적 (Lab이 보상 정책 운영)
- 자산화가 People팀 평가에 반영되는 구조 지향

### C4. 검수 실패 시 자산은 어떻게 되는가?

**A** (확정): 결과 3가지로 분리.

| 결과 | 처리 |
|---|---|
| 승인 | ③ 자산 데이터에 등재 |
| 반려 | 작성자 피드백 후 일정 기간 뒤 자동 폐기 (또는 수정 후 재제출) |
| 보류 | 추가 정보 요청. 30일 미보강 시 자동 폐기 |

### C5. ③ People Knowledge → ② 운영 표준 승급은 누가?

**A**: AX&CI Lab. 승급 hierarchy는:

```
③ 자산 데이터 작성/검수: Domain Experts (조직×도메인 권한 단위별)
② 운영 표준 자산 운영: AX&CI Lab
③ → ② 승급 결정: AX&CI Lab
```

## D. 진입과 강제

### D1. 강제 사용은 어떻게 작동하는가?

**A**: 부문 People팀장 선언에 의존. 1차 출시는 선언 + 도구 매력으로 자발적 수렴 유도.

**A**: Skill 정의·작성물은 차용. UX는 Work Card 채택. (별도 합의 필요)

## E. 운영 현실

### E1. 사외 LLM 호출 비용·할당량?

**A**: 사외 LLM은 사내 LLM 대비 토큰 여유가 있을 것으로 추정. 사내 LLM은 호출 제한이 빡빡함. 정확한 운영 정책은 AX팀 확인 필요.

### E2. 응답 지연 (차폐+호출+재매핑)?

**A**: 사용자 인내 시간 모니터 필요. 미정.

### E3. Tool 추가 요청 흐름?

**A** (확정):
```
HR 담당자 → Tool 요청 (양식)
   → AX&CI Lab 검토 (보안·중복·우선순위)
   → AX&CI Lab 구현
   → Tool 카탈로그 등록
   → Agent에서 호출 가능
```

## F. 네이밍 결정 사항

### F1. 데이터 3종 명칭 (확정)

| # | 결정 명칭 | 결정일 |
|---|---|---|
| ① | **운영 데이터 (Master)** | 2026-05-04 |
| ② | **표준 데이터 (Standards)** | 2026-05-04 |
| ③ | **자산 데이터 (Knowledge)** | 2026-05-04 |

### F2. 데이터 3종 후술

**① 운영 데이터 (Master)**
시스템이 갖춰야 할 최소 데이터. 로그인·권한 관리를 위한 사용자 식별 정보, 조직·개인·발령 등 플랫폼 자체 기능 수행에 필수적인 마스터 데이터. HR AX Platform 내부 운영 데이터로 적재한다.

**② 표준 데이터 (Standards)**
부서·전사 차원에서 공식적으로 운영되는 자료. 인사 제도, 규정, 업무 프로세스(L4/L5/L6), 표준 양식 등. AX&CI Lab이 운영하며 잦은 수정·생성을 직접 관리. 비정형 자료를 LLM이 이해하도록 각주·설명을 첨부하여 RAG·벡터화·학습에 사용.

**③ 자산 데이터 (Knowledge)**
HR 담당자의 사용 흔적이 누적되어 형성되는 People 조직 자산. Workbench의 Work Card 사용에서 발생한 노하우·판단·결정이 Tacit Candidate로 큐잉되고, Domain Experts 검수를 거쳐 자산 데이터에 등재. 조직×도메인 권한 단위와 공개 범위(개인 / 권한 단위 / 도메인 종방향 / 전사 표준)를 부착.

### F3. "Asset Registry" 표현 폐기 (확정)

별도 명사 없이 **"③ 자산 데이터(Knowledge)"** 자체가 검수 통과 자산의 집합. 행위는 **"자산 데이터에 등재"** 또는 **"검수형 자산화"**로 표현.

## G. 표현 정제

### G1. 사외 LLM과 도구 관계 표현

- ❌ "사외 LLM이 Tool/Skill/Agent에 접근"
- ⭕ "사외 LLM은 플랫폼이 정의한 Tool/Skill/Agent를 함수처럼 호출 요청만 하고, 실제 실행은 플랫폼 내부"
