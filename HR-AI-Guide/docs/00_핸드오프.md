# 인사부문 생성형 AI 활용 가이드 — 사내망 핸드오프 v0.2

> 이 문서는 사내망 환경에서 후속 작업자가 「HR 담당자용 생성형 AI 활용 가이드」를 이어서 작성·수정할 수 있도록 정리한 핸드오프 문서이다.  
> 과거 작업자명과 개인 호칭성 표현은 제외했다.  
> 작성일: 2026-05-12  
> 현재 산출 방향: **실제 배포용 검색형 HTML 가이드** + 필요 시 Word/요약본 보완

---

## 1. 프로젝트 목적

HR 담당자가 사내 승인 생성형 AI를 사용할 때, 입력 데이터의 위험도를 스스로 판단하고 안전하게 활용할 수 있도록 하는 실무 가이드를 만든다.

이 가이드는 기술적 안전장치가 완전히 정착되기 전까지, HR 담당자가 다음 기준으로 입력 여부를 판단하도록 돕는 임시·실무형 기준이다.

- **기밀 데이터**: AI 입력 금지
- **민감 데이터**: 담당자가 먼저 마스킹·익명처리·추상화한 후 조건부 사용
- **일반 데이터**: 사용 가능

핵심 메시지는 다음과 같다.

> **AI를 쓰지 말라는 것이 아닙니다. 안전하게 쓰시라는 것입니다.**

## 2. 현재까지 확정된 배포 방향

### 2.1 실제 배포물은 HTML 우선

기존 Word 초안은 정책문서·검토용 문서로 활용할 수 있으나, 실제 HR 담당자 배포물은 **검색형 HTML 가이드**가 더 적합하다.

HTML 가이드는 다음 기능을 갖는다.

1. 상단 검색창 제공
2. 사용자가 `주민번호`, `평가 코멘트`, `급여`, `면접 메모`, `M&A`, `임원 일정`, `노사협의회` 등 키워드를 입력
3. 하드코딩된 키워드/규칙 사전을 기반으로 RED/YELLOW/GREEN 분류 결과 제공
4. 처리 기준, 입력 전 제거·대체 항목, 예시, 안전한 프롬프트 제공
5. 판단이 모호하면 **AX&CI Lab 문의**로 fallback

중요: 현재 배포안은 LLM을 붙이지 않는 정적 HTML이다. 자연어 질문에 “답변처럼 보이는” 기능은 가능하지만, 이는 추론이 아니라 키워드·동의어·카테고리 매칭이다. 새로운 사례를 스스로 추론하지 않으므로 애매한 경우에는 자동 판정하지 않고 문의로 연결해야 한다.

### 2.2 후속 작업은 인터뷰식으로 진행

모호점 확인 시 한 번에 여러 질문을 던지지 않는다. 의사결정자가 follow-up하기 어렵지 않도록 **질문은 하나씩** 제시한다.

## 3. 회사 컨텍스트

### 3.1 기본 전제

- 회사는 상장사이다.
- HR 데이터는 개인정보, 평가·보상·채용·노무·조직·임원·M&A 관련 정보를 포함하므로 법적·평판상 위험이 높다.
- 본 가이드는 전사 「AI 활용 보안가이드」를 대체하지 않는다.
- 본 가이드는 전사 보안가이드 중 특히 제4호와 제5호를 HR 업무 맥락에서 실행하기 위한 실무 기준이다.
- 다만 별첨에는 제4호·제5호만이 아니라 전사 8대 수칙 전체를 포함해야 한다.

### 3.2 문의 채널

가이드 내 문의 채널명은 모두 다음으로 통일한다.

> **AX&CI Lab**

문구 예시:

> 판단이 어려운 경우 AI에 입력하기 전에 **AX&CI Lab**에 문의한다. 입력 후 사후 보고하는 것보다 입력 전 확인하는 것이 원칙이다.

## 4. HR 데이터 10대 분류

정보보호센터와 협의 완료된 HR 데이터 10대 분류를 그대로 사용한다. 본 가이드는 등급 자체를 변경하지 않는다.

| # | 카테고리 | 등급 | 예시 | AI 입력 기준 |
|---:|---|---|---|---|
| 1 | 노조/협의회 관련 | RED / 기밀 | 조합 협의 안건, 협의회 현황, 단체교섭 관련 내용 | 입력 금지 |
| 2 | 고유식별정보 DB | RED / 기밀 | 주민등록번호, 외국인등록번호, 여권번호 등 | 입력 금지 |
| 3 | 임원/조직 | YELLOW / 민감 | 임원 처우·근황, 조직개편, 임원 일정 | Level 3 맥락 추상화 후 조건부 사용 |
| 4 | M&A | YELLOW / 민감 | M&A 계획, 실사자료, PMI 인력방안 | Level 3 맥락 추상화 후 조건부 사용 |
| 5 | 인사 기본 | YELLOW / 민감 | 성명, 학력, 가족관계, 경력 | Level 2 익명처리 후 조건부 사용 |
| 6 | 성과/평가 | YELLOW / 민감 | 업적평가, 업무목표, 평가 코멘트 | Level 2 익명처리 후 조건부 사용 |
| 7 | 핵심/양성/리텐션 | YELLOW / 민감 | 핵심등급, 양성이력, 리텐션 대상 | Level 3 맥락 추상화 후 조건부 사용 |
| 8 | 보상/근태/복리후생 | YELLOW / 민감 | 월급여, 연봉, 출퇴근 기록, 휴직·복리후생 | Level 2 익명처리 후 조건부 사용 |
| 9 | 채용 | YELLOW / 민감 | 이력서, 면접 점수, 평판조회, 후보자 정보 | Level 2~3 적용 후 조건부 사용 |
| 10 | 총무 | YELLOW / 민감 | 사내 인프라, 차량, 좌석, 출입 관련 정보 | Level 1~2 적용 후 조건부 사용 |

### 4.1 잘못된 단순화 방지

다음 표현은 피한다.

> “기밀 2개를 제외하면 전부 민감이다.”

이 표현은 과도하게 단순화되어 일반 질의의 위치를 흐릴 수 있다. 대신 다음 구조를 사용한다.

- **RED / 기밀**: 10대 분류 중 기밀로 지정된 2개 영역. AI 입력 금지.
- **YELLOW / 민감**: 10대 분류 중 민감으로 지정된 HR 데이터. 입력 전 담당자가 직접 가공.
- **GREEN / 일반**: 공개자료 또는 회사·개인·조직 식별정보가 없는 범용 업무 질의.

또한 “비공개 내부정보”라는 표현은 담당자가 판단하기에 애매할 수 있으므로 본문에서 남용하지 않는다. 본문은 **10대 분류표 + 키워드 + 예시 + 체크리스트** 중심으로 설계한다.

## 5. RED / YELLOW / GREEN 3색 라벨

가이드 최상단에는 다음 도식을 제공한다.

```text
AI에 입력하려는 데이터가 있다
        ↓
[1] RED / 기밀 데이터인가?
    - 노조/협의회 관련 정보
    - 주민등록번호·외국인등록번호·여권번호 등 고유식별정보 DB 또는 이에 준하는 고유식별정보
        ↓
    YES → 입력 금지
    NO  → 다음 단계

[2] YELLOW / 민감 데이터인가?
    - HR 10대 분류상 민감 항목
    - 개인식별정보, 평가·보상·채용·조직·임원·M&A 등 보호가 필요한 HR 업무 데이터
        ↓
    YES → 담당자가 직접 마스킹·익명처리·추상화 후 조건부 사용
    NO  → 다음 단계

[3] GREEN / 일반 데이터인가?
    - 공개자료
    - 회사·개인·조직 식별정보가 없는 일반 업무 질의
        ↓
    YES → 사용 가능
    판단 불가 → AX&CI Lab 문의
```

### 5.1 본문 도식 표현

| 라벨 | 의미 | AI 입력 기준 |
|---|---|---|
| RED / 기밀 | AI에 입력해서는 안 되는 정보 | 입력 금지 |
| YELLOW / 민감 | HR 10대 분류상 민감하거나 보호가 필요한 HR 데이터 | 입력 전 직접 마스킹·익명처리·추상화 후 조건부 사용 |
| GREEN / 일반 | 공개자료 또는 내부 식별정보 없는 일반 질의 | 사용 가능 |

| 체크 | 질문 | 해당 시 처리 |
|---:|---|---|
| 1 | 주민등록번호, 외국인등록번호, 여권번호 등 고유식별정보가 있는가? | RED. 입력 금지 |
| 2 | 노조/협의회 안건, 가입, 발언, 협상 내용이 있는가? | RED. 입력 금지 |
| 3 | 이름, 사번, 연락처, 이메일, 사진 등 개인을 바로 알아볼 정보가 있는가? | YELLOW. 제거·대체 후 사용 |
| 4 | 부서, 직책, 연차, 학교, 전 직장, 평가등급 등이 조합되어 특정인을 추정할 수 있는가? | YELLOW. 익명처리·집계화 후 사용 |
| 5 | 평가, 보상, 채용, 핵심인재, 임원/조직, M&A 등 HR 10대 분류에 해당하는가? | YELLOW. 분류별 Level 적용 후 사용 |
| 6 | AI에게 채용·평가·보상·징계 등 인사결정 자체를 요청하는가? | 데이터 등급과 무관하게 금지 |
| 7 | 위 항목 판단이 어렵거나 여러 항목이 섞여 있는가? | 입력 중단 후 AX&CI Lab 문의 |

## 7. YELLOW 데이터 처리 3단계

“약/중/강”이라는 표현만으로는 실무자가 무엇을 해야 하는지 알기 어렵다. 따라서 다음 명칭을 사용한다.

| 기존 표현 | 실무형 명칭 | 핵심 행동 |
|---|---|---|
| 약 | Level 1. 기본 마스킹 | 직접 식별자를 제거·대체한다 |
| 중 | Level 2. 익명처리 + 조합식별 제거 | 직접 식별자뿐 아니라 조합하면 특정되는 정보도 줄인다 |
| 강 | Level 3. 맥락 추상화 | 사람·조직·회사·시점·금액·프로젝트 맥락까지 일반화한다 |

### 7.1 Level 1. 기본 마스킹

- 적용: 총무성 정보, 일반 사내 문의, 단순 문장 다듬기 등
- 처리: 이름, 사번, 연락처, 이메일, 생년월일 등 직접 식별자 제거
- 예시: `[직원명]에게 복지포인트 안내 메일 작성` → `직원 A에게 복지포인트 안내 메일 작성`

### 7.2 Level 2. 익명처리 + 조합식별 제거

- 적용: 인사 기본, 평가, 보상, 근태, 채용 등
- 처리: 이름뿐 아니라 부서·직책·연차·학교·전 직장·평가등급·정확한 금액 등 조합 식별 요소도 축소
- 예시: `[부서명] [직책] [직원명], 평가등급 [등급], 연봉 [정확한 금액]` → `직군 단위 직원 A, 평가군 상위, 보상 구간`

### 7.3 Level 3. 맥락 추상화

- 적용: 임원/조직, M&A, 핵심인재/리텐션, 임원 보상, 미공개 조직개편 등
- 처리: 개인명뿐 아니라 회사명, 조직명, 임원명, 일정, 금액, 프로젝트명, 거래 상대방, 시점까지 일반화
- 예시: `[사업부명] 조직개편으로 [팀명] 통합, [임원명] 담당` → `특정 조직 운영 변경에 대한 일반 커뮤니케이션 문안`

## 8. 프롬프트 작성의 핵심 원칙

가이드에 반드시 다음 원칙을 강하게 넣는다.

> **민감정보를 입력한 뒤 “답변에 포함하지 말라”고 지시하는 것은 비식별화가 아니며, 허용되지 않는다. 민감정보는 AI 입력 전에 반드시 제거·대체·추상화해야 한다.**

사용자에게 기억시키기 쉬운 문구:

> **AI에게 숨기라고 하지 말고, AI가 볼 수 없게 만든 뒤 입력한다.**

### 8.1 잘못된 패턴

```text
[직원명], [부서명], [정확한 연봉] 관련 내용입니다.
단, 답변에는 [직원명], [부서명], [정확한 연봉]을 포함하지 마세요.
```

이 방식은 금지한다. 답변에 포함하지 말라고 해도, 이미 프롬프트에 입력된 순간 입력 행위가 발생한다.

### 8.2 올바른 패턴

```text
1. 원문 데이터는 AI에 넣지 않는다.
2. 사람이 먼저 원문에서 식별자·조합식별자·맥락 정보를 제거한다.
3. AI에는 가공된 값만 넣는다.
4. 프롬프트에는 “어떤 실제값을 빼라”가 아니라 “어떤 형식으로 작성하라”고 요청한다.
```

권장 예시 구조:

| 단계 | 내용 |
|---|---|
| 원문 데이터 | AI에 입력하지 않음 |
| 사람이 먼저 가공한 입력값 | 직원 A / 직군 단위 / 구체 금액 제거 / 평가등급 제거 |
| AI 입력 프롬프트 | “아래는 특정 개인을 식별할 수 없도록 가공한 문장입니다. 건설적이고 균형 잡힌 표현으로 다듬어 주세요.” |

## 9. 안전한 프롬프트 예시 설계 원칙

HTML 검색 결과 카드에는 다음 순서로 예시를 제공한다.

1. 이 데이터가 왜 RED/YELLOW/GREEN인지
2. AI에 넣으면 안 되는 원문 요소의 **범주**
3. 입력 전 담당자가 직접 바꿔야 하는 방식
4. AI에 입력 가능한 가공된 프롬프트
5. 판단이 어려울 때 문의 채널

실제 민감값을 예시 프롬프트 안에 쓰지 않는다. 예시는 `[직원명]`, `[부서명]`, `[정확한 금액]` 같은 placeholder 또는 “직원 A”, “직군 단위”, “보상 구간”처럼 가공된 값만 사용한다.

## 10. 법적 특수성 반영 방식

본문에는 법적 특수성을 짧게 설명하고, 별첨에는 자세히 설명한다.

### 10.1 본문 요약 문구 예시

> HR 데이터는 개인정보, 평가·보상·채용 정보, 노무 관련 정보, 임원·조직·M&A 정보 등으로 구성되어 있어 일반 업무자료보다 법적·평판상 위험이 크다. 특히 개인정보, 민감정보, 고유식별정보는 개인정보보호법상 처리 제한과 안전조치 의무가 적용될 수 있으며, 상장사의 미공개 조직·M&A 정보는 자본시장법상 미공개중요정보 이슈와 연결될 수 있다. 따라서 HR 담당자는 AI 활용 전 입력 데이터의 등급을 확인하고, RED 데이터는 입력하지 않으며, YELLOW 데이터는 입력 전에 직접 마스킹·익명처리·추상화해야 한다.

## 11. 전사 AI 활용 보안가이드 8대 수칙

다음 원문은 변경하지 않는다. 본 가이드는 이를 HR 업무 맥락에서 구체화한다.

```text
1. 회사에서 승인된 기업용 AI 서비스만 업무에 활용
   1) AI지원부서(AIX)가 공식적으로 사용 승인한 기업용 AI 서비스만 사용
   2) 인가되지 않은 개인용/무료 외부 AI 서비스는 사내에서 활용 제한

2. 회사에서 승인한 기업용 AI 서비스 접속 시 개인 계정 사용 제한
   1) 공식적으로 사용 승인된 기업용 AI 서비스라 하더라도 개인 계정으로 사용 제한
   2) 사내에서는 반드시 회사에서 제공한 기업용 계정만 사용

3. 비인가 브라우저 확장 프로그램 및 AI 에이전트 설치 및 사용 제한
   1) PC 브라우저 내 출처 불명의 AI 확장 프로그램(크롬 익스텐션 등)을 임의 설치 제한
   2) PC 접근 및 사내 시스템 권한을 요구하는 외부 AI Agent 무단 설치 및 사용 제한

4. 기밀(극비) 데이터 입력 제한
   1) 국가 핵심기술 등 회사에서 지정한 기밀 데이터는 승인된 기업용 AI 서비스 사용 제한
   2) 기업용 계정 발급받지 못한 기밀 데이터 취급자는 타인의 계정을 도용/대여 AI 활용 제한

5. 텍스트 입력 전 개인정보 및 민감정보 비식별화 확인
   1) 질의(프롬프트) 내 고객/임직원 개인정보 및 민감정보(서버 계정정보 등) 입력 제한
   2) 업무를 위해 입력이 필요할 경우에는 원문 입력 대신 가명처리 또는 비식별 처리 후 활용

6. 의도하지 않은 정보의 업로드 방지를 위한 첨부 파일 사전 검수
   1) AI 서비스에 파일 업로드 전 엑셀의 '숨겨진 시트', '필터 영역' 등 불필요 데이터 확인

7. Agent 사용 시 중요 동작 자동실행 설정 제한/사용자 승인(Human-In-The-Loop) 준수
   1) 이메일 발송, DB 수정, 코드 수정/반영 등 Agent의 중요 동작은 단독/자동 실행 설정 제한
   2) 반드시 사용자가 최종 수행 내역 확인 후 직접 실행버튼을 클릭(HITL)

8. AI 결과물 활용 전 사실 여부 및 저작권 교차 검증
   1) AI 생성 결과물 내 허위 정보(환각) 포함 가능성 인지하여 비판적 검토 수행
   2) 결과물 실무 적용 전, 직접 사실 확인 및 저작권/라이선스 침해 여부 교차 검증
```

## 13. 인사결정 위임 금지

데이터 등급과 별도로, AI에게 다음 행위를 요청해서는 안 된다.

- 지원자 합격·불합격 판단
- 임직원 평가등급 산정
- 보상 인상률 또는 성과급 배분 결정
- 징계 수위 결정
- 핵심인재, 승계후보, 리텐션 대상 선정
- 특정 직원을 비교·순위화하여 인사결정을 추천하게 하는 행위

허용되는 범위는 다음과 같다.

- 식별정보가 제거된 문장의 표현 다듬기
- 일반적인 안내문 초안 작성
- 평가·보상·채용 제도 운영 시 고려사항 정리
- 사람이 이미 결정한 내용을 설명하는 커뮤니케이션 초안 작성
- 공개자료 기반 리서치 요약

## 14. 검색형 HTML 카드 설계

검색 결과는 B안, 즉 실무 안내형으로 제공한다.

표준 카드 구조:

```text
[YELLOW / 민감 데이터]
카테고리: 성과/평가
처리 기준: 입력 전 익명처리 + 조합식별 제거
권장 처리 수준: Level 2

왜 민감한가
평가 정보는 임직원 개인에게 직접 영향을 주는 판단 정보이며, 이름을 제거해도 부서·직책·연차·등급 조합으로 특정될 수 있다.

AI에 넣으면 안 되는 원문 요소
- 이름, 사번, 이메일, 연락처
- 부서명, 직책, 연차
- 평가등급, 점수, 순위
- 평가자 정보, 비교 대상 직원 정보

입력 전 사람이 바꿔야 하는 방식
- 직원명 → 직원 A
- 부서명 → 직군 또는 기능 단위
- 평가등급 → 평가군 또는 경향
- 점수·순위 → 구간 또는 일반 표현

AI에 입력 가능한 프롬프트
아래는 특정 개인을 식별할 수 없도록 가공한 평가 피드백 문장입니다.
문장을 건설적이고 균형 잡힌 표현으로 다듬어 주세요.

[가공된 문장 입력]
```

## 17. Memory 참고 내용 반영

이전 개인화 기억에서 확인된 HR AI 거버넌스 참고 요소는 다음과 같다. 상세 정의가 별도 파일로 확인된 것은 아니므로, 본 가이드에서는 부록의 참고 프레임으로만 활용한다.

- HR AI 역할 4가지: 증강자, 스튜어드, 인테그레이터, 이네이블러
- 3×3 개입 매트릭스
- 민감정보·저작권 점검
- RBAC 기반 접근통제
- HITL 기준
- 결과 모니터링
- 에스컬레이션 경로

HTML 별첨에는 이를 “운영 거버넌스 참고” 정도로 요약한다. 본문 핵심 판단 기준을 흐리지 않도록, 세부 프레임은 보조 자료로만 둔다.

## 18. 남은 의사결정 사항

다음 항목은 추가 확인이 필요하다. 확인은 한 번에 하나씩 진행한다.

1. 법적 특수성 별첨의 상세 수준: 조항 번호 중심인지, 조항별 실무 해석까지 넣을지
2. HTML 검색 카드의 초기 키워드 범위: HR 10대 분류별 10~20개 키워드까지 확대할지
3. 사고 발생 시 대응 절차: AX&CI Lab 단일 문의 후 내부 라우팅인지, 정보보호센터/개인정보/법무 병행 표기인지
4. 징계·고충·노무 사안의 데이터 등급: YELLOW 고위험으로 둘지, 특정 경우 RED로 별도 지정할지
5. 실제 배포 디자인: 사내 포털 embedded HTML인지, 단독 HTML 파일인지

## 19. 후속 작성 원칙

- 회사가 이미 정한 HR 10대 분류는 변경하지 않는다.
- M&A, 임원/조직, 핵심/양성/리텐션을 임의로 RED로 상향하지 않는다. YELLOW로 유지하되 Level 3 맥락 추상화를 적용한다.
- “비공개 내부정보”처럼 추상적이고 애매한 표현보다 키워드·예시·처리 단계로 안내한다.
- 민감정보를 프롬프트에 넣은 뒤 “출력하지 말라”고 지시하는 예시는 만들지 않는다.
- RED 데이터는 원칙적으로 가공 후 조건부 사용 예시를 제공하지 않는다. 실제 내용이 없는 일반 원칙 질의만 허용한다.
- 문의 채널은 AX&CI Lab으로 통일한다.
- 후속 확인 질문은 한 번에 하나씩 제시한다.
