### 3.2 첫 메시지 — 컨텍스트 인지 확인

다음 메시지를 복붙하여 시작:

```
안녕. 이전 챗에서 HR 생성형 AI 활용 가이드를 작성하던 작업을 이어서 진행한다.

먼저 다음을 순서대로 읽고 현재 상태를 파악해줘:
1. docs/00_핸드오프.md — 작업 히스토리 전체
2. docs/01_5대원소_정의.md — 합의된 원소 정의
3. docs/05_철회된_권고.md — 반복하면 안 되는 권고들
4. data/HR_데이터_마스터_v0.3.xlsx — 현재 데이터 마스터 (Lab 피드백 대기 중)
5. deliverables/guide_html/ 안의 HTML — 현재 배포물 v0.2

다 읽고 나면 다음을 알려줘:
(1) 현재까지 합의된 내용 5줄 요약
(2) 다음 작업 후보 (CLAUDE.md §「다음 작업 후보」 기준)
(3) 권고하는 우선순위와 이유

이후 내가 우선순위를 결정하면 그 작업부터 진행한다.
```

→ Claude Code는 파일을 직접 읽고 컨텍스트를 *능동적으로* 흡수합니다. 이게 chat과 결정적으로 다른 점.

## 💡 4. Claude Code 작업 시 활용 팁

### 4.1 .claude/ 폴더 활용

Claude Code는 프로젝트별로 `.claude/` 폴더에 메모를 저장합니다. 다음과 같이 활용:

```
.claude/
├── memories/                # 작업 중 발견·결정사항 누적
└── commands/                # 자주 쓰는 명령어 단축
    └── update-handoff.md    # "핸드오프 문서 v0.X로 업데이트해줘" 같은 자주 쓰는 작업
```

### 4.2 매 세션 종료 시 핸드오프 갱신

작업 종료 직전 이런 명령:

```
오늘 작업한 내용을 docs/00_핸드오프.md에 반영해줘.
- 새로 결정된 사항
- 변경된 산출물 상태
- 다음 세션 시작점

업데이트 후 git commit 메시지도 작성해줘.
```

### 4.3 엑셀 직접 편집

Claude Code는 `openpyxl`을 직접 호출할 수 있어, Lab 피드백을 받으면 엑셀에 *직접 반영* 가능. 챗에서는 매번 파일을 다시 만들었지만, Claude Code에서는 *부분 수정*이 가능합니다.

### 4.4 HTML 카드 일괄 업데이트

엑셀 마스터의 `Lab원 검토` 컬럼이 채워지면, Claude Code에게:

```
data/HR_데이터_마스터_v0.3.xlsx 에서 「확정 여부」가 ✅로 변경된 행들을 읽어,
deliverables/guide_html/ 안의 HTML 카드를 일괄 업데이트해줘.
```

→ 챗에서는 일일이 짚어야 했던 작업이 *자동화*됩니다.

## ⚠️ 5. 이관 시 주의사항

### 5.1 Claude Code는 "백지에서 시작"

본 챗에서의 *모든 합의 사항*은 J의 머릿속에만 있고, Claude Code의 인스턴스는 그것을 모릅니다.

→ `CLAUDE.md`와 `docs/` 폴더가 *유일한 컨텍스트 전달 수단*. 핵심 내용이 빠지면 새 인스턴스는 빗나간 권고를 반복합니다.

### 5.2 메모리 누락 사항 확인

본 챗의 Claude 메모리에는 다음이 저장되어 있습니다 (Claude Code는 이걸 모름):

- J 회사가 상장사이며 자본시장법 적용
- 정보보호센터의 "기밀 최소화" 전략
- 기밀 데이터 취급자에게는 외부 LLM 차단
- 5대 원소 합의 v0.2 → v0.3 진척사항
- 문의 채널명 AX&CI Lab

→ 이 모두가 `CLAUDE.md`와 `docs/` 에 들어가 있는지 확인.

### 5.3 본 챗의 데이터 마스터 64행은 *초안*

Lab 피드백을 받지 않은 상태이므로, Claude Code 첫 세션에서는 이 점을 명확히 인지시킬 것:

> "엑셀의 ✅ 표시는 본 챗 Claude의 *제안*이고, 실제 Lab 합의는 아직 받지 않음. Lab 피드백을 받은 후에야 진짜 확정."

## ❓ 7. FAQ

### Q. 본 챗을 계속 쓰면 안 되는가?

가능합니다. 다만 다음 경우엔 Claude Code가 명확히 유리:
- 토큰이 자주 부족하다고 느낌 → Claude Code는 context window 관리 우수
- 같은 파일을 여러 번 수정 → 디스크 직접 편집이 훨씬 빠름
- git으로 버전 관리하고 싶음
- Lab 피드백을 받아 *부분 수정*해야 함 (전체 재생성 X)

### Q. CLAUDE.md를 매번 업데이트해야 하나?

큰 합의가 추가될 때마다 업데이트하는 게 안전합니다. 매 세션 끝날 때 Claude Code에게:

> "오늘 세션에서 추가·변경된 합의 사항이 있으면 CLAUDE.md에 반영해줘."

### Q. 여러 LLM에 같은 컨텍스트를 줄 수 있나?

가능합니다. `docs/` 폴더의 .md 파일들을 다른 LLM의 chat에 통째로 붙여넣으면 됩니다.
`CLAUDE.md`는 Claude Code 전용이지만, 내용 자체는 다른 LLM에도 유용.

### Q. 본 챗에서 결정한 정보가 누락된 게 있는지 걱정된다.

Claude Code 첫 세션에서 이렇게 물어보세요:

> "지금까지 읽은 모든 문서에서, 「당신이 알아야 할 정보가 빠진 부분」이 있다면 알려줘.
> 특히 J 회사 컨텍스트나 합의 사항 중 모호하거나 누락된 게 있는지."

Claude Code가 발견한 빈틈을 J가 직접 보강하면 됩니다.
