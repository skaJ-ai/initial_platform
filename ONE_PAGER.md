## 한 줄

> **HR 데이터가 AI Tool로 가기 전에 권한 확인·등급 분류·비식별화를 거치게 하고, 사용 흔적을 People 조직 자산으로 축적하는 HR 특화 Platform.**

---

## 왜 HR 특화 Platform인가

전사 공통 AI Platform, 사내 LLM(GAUSS), 사외 LLM(Chat GPT / Gemini 등), 사내 AI Tool은 모델·인프라로 활용할 수 있다. 그러나 원문 HR Data와 HR Knowledge는 HR 도메인 권한, 민감도, 검수 책임이 붙어야 하므로 공통 Platform이나 사외 LLM에 그대로 올릴 수 없다.

| 이유 | HR AX가 맡아야 하는 것 |
|---|---|
| **보안** | AI Tool 앞단의 안전 게이트. 기밀/민감 HR 데이터 분류, 사외 송출 차단, 비식별화, audit |
| **축적** | Light user가 직접 RAG/DB를 관리하지 않아도 Work Card 사용 흔적을 자산 데이터로 누적 |
| **권한** | 조직×도메인 권한 단위로 DW, 표준 데이터, 자산 데이터 접근을 필요한 만큼만 허용 |

---

## 3대 축

### ① 보안 — 사외 LLM을 사용하기 위한 안전한 관문

데이터 등급(기밀 / 민감 / 일반)에 따라 처리 경로가 분기됨.
민감 등급은 비식별화 게이트가 사외 송출 시 차폐, 응답 수신 시 사내 재매핑.
매핑 테이블은 사내에만 보관. 외부엔 차폐된 업무 맥락만 전달.
원문 HR Data/Knowledge는 전사 공통 AI Platform에 올리지 않음.

**메시지**: "**민감 정보는 사외로 나가지 않습니다**"

### ② 축적 — Light user도 쓰기만 하면 자산화

Work Card 사용에서 누적된 노하우·판단이 Tacit Candidate로 큐잉되고, Domain Experts 검수를 거쳐 ③ 자산 데이터에 등재.
HR 담당자는 AI 빌더가 아니다. 사업부/도메인별 RAG를 직접 만들거나 로컬 DB를 관리하게 두면 세션 단위로 초기화된다. HR AX는 표준 데이터·자산 데이터·Skill을 Work Card에 자동 부착한다.

**메시지**: "**쓸수록 People 조직의 지식 자본이 됩니다**"

### ③ 권한 — 필요한 사람에게 필요한 만큼만

DW의 원천 HR 데이터, 플랫폼의 운영/표준/자산 데이터는 조직×도메인 권한 단위로 접근 가능 범위가 달라진다. 자산 데이터 공개 범위는 작성자가 제안하고 검수자가 확정한다. 개인 → 조직×도메인 권한 단위 → 도메인 종방향(부문) → 전사 표준 단계로 넓어질수록 추가 검수가 붙는다.

**메시지**: "**접근 권한이 있는 데이터와 Knowledge만 AI가 활용합니다**"

## 데이터 등급 × 처리 경로

| 등급 | 처리 | 모델 |
|---|---|---|
| **기밀** | 사외 송출 금지 | 사내 LLM (GAUSS, 로그 미저장) |
| **민감** | 비식별화 게이트 후 외부 | 사외 LLM (Chat GPT / Gemini 등) |
| **일반** | 그대로 외부 | 사외 LLM |

**자동 분류 3계층**: 조직×도메인 기본 등급 → PII 패턴 매칭 → 데이터 민감도 분류 Agent(GAUSS 기반). 분류 단계는 민감 데이터가 들어올 수 있으므로 사외 LLM을 절대 호출하지 않는다.

**사용자 조정 원칙**: 사용자는 등급 상향만 가능. 상향은 즉시 적용하고 audit log, 권한 단위 Domain Expert 통지, 분류표 v0 갱신 트리거로 남긴다.

## 단순 챗 + Skill만으로 안 되는 이유

> "사외 LLM에 Skill 라이브러리만 주면 끝 아닌가?" 라는 질문에 대한 답.

| 요구사항 | 단순 챗 + Skill 라이브러리 | HR AX Platform |
|---|---|---|
| HR 데이터 등급(기밀/민감/일반) 자동 분류 | ❌ Skill엔 등급 개념 없음 | ⭕ 조직×도메인 + PII 패턴 + 사내 분류 Agent |
| 기밀 데이터를 외부에 안 보내기 | ❌ Skill에 데이터를 넣는 순간 사외 송출 | ⭕ 기밀은 사내 LLM(GAUSS, 로그 미저장) 전담 |
| 민감 데이터 비식별화 후 외부 사용 | ❌ Skill엔 차폐 로직 없음 | ⭕ 비식별화 게이트 + 사내 재매핑 |
| 조직×도메인별 다른 차폐 정책 | ❌ Skill md엔 정책 못 담음 | ⭕ 권한 단위별 차폐 정책표 |
| 같은 권한 단위 담당자 자산 공유 | ❌ 콜드 스타트 매번 처음부터 | ⭕ Workbench가 자산 retrieval |
| 부문이 산하 사업부 자산 열람 | ❌ Skill엔 조직 계층 개념 없음 | ⭕ 도메인 종방향 승급으로 자동 |
| 비정형 → 정형/Skill 승격 | ❌ Skill은 정적 md, 진화 안 함 | ⭕ 반복 패턴이 신규 L6 또는 Skill 후보로 포착 |
| Domain Experts 검수 워크플로 | ❌ 사용자가 알아서 | ⭕ 승인/반려/보류 3분기 검수 |
| 매핑 사내 보관 / Audit | ❌ 사외 LLM 책임 | ⭕ Application-side audit, 매핑 사내 |
| 표준 데이터 자동 적용 | ❌ Skill에 매번 복붙 | ⭕ Lab 운영 표준 데이터 자동 RAG |
| 인사 DB 권한별 접근 차등 | ❌ Skill엔 RBAC 없음 | ⭕ Workbench 권한이 DB 접근 게이트 |
| 강제 사용 정당성 | ❌ "그냥 Skill 좋네" | ⭕ "안전 관문 통과 안 하면 누설 사고" |

**결론**: 사외 LLM + Skill 라이브러리는 **점**. 우리 Platform은 그 점들을 **조직×도메인 권한 단위 + Workbench + 검수 파이프라인**으로 묶는 격자. 점은 격자가 없으면 흩어진다.

## 데이터 3종 (플랫폼이 적재)

### ① 운영 데이터 (Master)
시스템이 갖춰야 할 최소 데이터. 로그인·권한 관리를 위한 사용자 식별 정보, 조직·개인·발령 등 플랫폼 자체 기능 수행에 필수적인 마스터 데이터. HR AX Platform 내부 운영 데이터로 적재한다.

### ② 표준 데이터 (Standards)
부서·전사 차원에서 공식적으로 운영되는 정형·권위 자료. 인사 제도, 인사 기준(근태·평가·보상 등), 업무 프로세스(L4/L5/L6), 표준 양식으로 한정. AX&CI Lab이 운영하며 사람용 문서와 LLM RAG 양쪽에서 사용.

### ③ 자산 데이터 (Knowledge)
HR 담당자의 사용 흔적이 누적되어 형성되는 People 조직 자산. 사용 흔적 노하우, 판단 메모, 결정 narrative, 예외 케이스가 Tacit Candidate로 큐잉되고, Domain Experts 검수를 거쳐 자산 데이터에 등재. 조직×도메인 권한 단위와 공개 범위(개인 / 권한 단위 / 도메인 종방향 / 전사 표준)를 부착. 다수는 자산 데이터에 잔류하고, 일부만 표준 데이터 또는 Skill로 승격된다.

## Function Calling 패턴 (사외 LLM과 도구의 관계)

```
1. 플랫폼이 사외 LLM에 Tool 목록 + Skill 컨텍스트 묘사 전달
   (Tool 코드 자체는 안 보냄)
2. 사외 LLM이 "Tool X를 인자 Y로 호출" 의도 반환
3. 플랫폼이 Tool 실제 실행 (사내)
4. 결과를 비식별화 게이트 통과 후 사외 LLM에 컨텍스트로
5. 사외 LLM이 최종 응답 생성 → 사내 재매핑 → 사용자
```

**핵심**: 사외 LLM은 의도만 표현. 실제 실행·데이터 노출 통제는 플랫폼이 전담.

## HR Data 및 권한 관리 체계

**X축 (조직)**: AX&CI Lab(admin) / 부문(본사) / 사업부 8개(VD, DA, MX, NW, 의료기기, SR, 생기연, CDO)
**Y축 (도메인 12개)**: 교육, 노사, 보상·근태, 인력운영, 임원조직, 제도, 채용, 총무, 해외인사, DEI, 조직문화, 집단지성

조직×도메인 조합이 Workbench의 기본 **권한 단위**다. 각 권한 단위마다 **DW 접근 범위 / 플랫폼 데이터 접근 범위 / 기본 등급(기밀·민감·일반) / 차폐 정책 / Domain Experts**가 부착된 **분류표 v0**가 필요하다.

```
[사용자]
   ↓
[조직 역할 + 담당 도메인 확인]
   ↓
[권한 단위 = 조직 × 도메인]
   ↓
┌──────────────────────┬────────────────────────────┐
│ DW                   │ 원천 HR 데이터 통합 저장소  │
│ HR AX Platform        │ ① 운영 ② 표준 ③ 자산 데이터 │
└──────────────────────┴────────────────────────────┘
   ↓
[필요한 만큼만 검색/RAG/Tool 실행]
   ↓
[분류 Agent → 라우팅 → 사내 LLM(GAUSS) 또는 사외 LLM]
```

원문 HR Data와 검수 통과 전후 HR Knowledge는 공통 AI Platform에 적재하지 않는다. 사외 LLM/전사 AI Tool은 HR AX가 허용한 차폐 컨텍스트 또는 호출 결과만 받는다.

## AI Tool 연계 원칙

| AI Tool | 역할 | 연결 원칙 |
|---|---|---|
| 사내 LLM (GAUSS) | 기밀 처리, 분류 보강, 사내 임베딩 | LLM 자체 로그 미저장. 원문 HR 데이터 처리 가능. HR AX application-side audit만 기록 |
| 사외 LLM (Chat GPT / Gemini 등) | 일반/비식별화 후 복잡 추론 | 원문 HR Data/Knowledge 직접 접근 금지 |
| 사내 AI Tool | 공통 AI 기능 보조 | HR AX의 권한·분류·차폐 정책을 통과한 경우에만 제한 연계 |

※ 사외 LLM은 원문 HR Data/Knowledge를 직접 보지 않는다. HR AX가 권한 확인, 비식별화, 감사 로그를 거친 차폐 컨텍스트만 전달한다.

사외 LLM/전사 AI Tool은 Tool 실행 권한을 직접 갖지 않는다. 호출 의도만 표현하고, HR AX가 권한 확인 후 내부에서 실행한다.

## 검수 결과 3분기

| 결과 | 처리 |
|---|---|
| **승인** | ③ 자산 데이터에 등재. 작성자·검수자 양쪽 실적 |
| **반려** | 작성자 피드백. 일정 기간 후 자동 폐기 또는 수정 후 재제출 |
| **보류** | 추가 정보 요청. 30일 미보강 시 자동 폐기 |

자산화 = Domain Experts 실적. Lab이 검수자 보상 정책 운영.

## 공개 범위 확정 원칙

작성자가 카드 종료 시 공개 범위를 제안하고, 검수자가 승인 시 확정한다. 디폴트는 조직×도메인 권한 단위 한정. 좁히는 것은 검수자 단독으로 가능하지만, 넓히는 것은 단계적 추가 검수가 필요하다.

| 확대 경로 | 추가 검수 |
|---|---|
| 권한 단위 → 도메인 종방향 | 부문 Domain Expert |
| 도메인 종방향 → 전사 표준 | AX&CI Lab |
