# Context — 외부 환경

> v3 가설이 도출된 외부 조건들. 이 조건이 바뀌면 가설도 재검토해야 함.

---

## 1. 사외 LLM/사내 AI Tool 활용 환경 도래

### 1.1 사실관계

- 2026-07부터 사내망 안에서 사외 LLM(Chat GPT, Gemini 등)과 일부 사내 AI Tool 활용 환경이 넓어짐
- 모델뿐 아니라 워크스페이스/지식 업로드/툴콜링 등 제품 단위
- 도입 주체: AX팀 (People팀과 무관)
- 사내망 동거인이라 "데이터 격리"만으로는 차별 안 됨

### 1.2 사외 LLM/AI Tool이 가져오는 것 (추정)

대부분의 엔터프라이즈급 AI Tool은 다음을 포함:
- SSO/SCIM 연동
- 권한 통제 (RBAC, 프로젝트 단위)
- Audit log
- Retention 0 옵션
- SOC2 / ISO27001 / HIPAA 준수
- 워크스페이스 / 지식 업로드 / 툴콜링 / 검색 통합

→ 일반 거버넌스(권한/감사/데이터 retention)는 일부 같이 들어올 수 있음.
→ 그러나 원문 HR Data/Knowledge를 공통 AI Platform이나 사외 LLM에 직접 올리는 것은 별도 문제.
→ 살아남는 차별은 **AI Tool 앞단에서 HR 도메인 보안·축적·권한을 통제하는 Platform 영역**.

### 1.3 100% 안전한가? — 아니다 (누수 채널 5종)

| # | 채널 | 의미 |
|---|---|---|
| 1 | 추론 시 일시 캐시/메모리 잔존 | retention 0이라도 처리 중 메모리에 떠 있음. 격리는 vendor 보장 의존. |
| 2 | 벤더 운영 인력 접근 | 인시던트 / 디버깅 시 일시적 접근 (NDA + 감사로 통제하지만 0은 아님) |
| 3 | 컴플라이언스 보존 의무 | 일부 국가/산업은 법적으로 일정 기간 로그 보존 강제 |
| 4 | 프롬프트 인젝션 누수 | 외부 공격으로 모델이 자기 컨텍스트를 토해내게 만들 수 있음 |
| 5 | 국가 단위 데이터 요청 | 미국 벤더는 CLOUD Act 등으로 데이터 제출 의무 가능 |

→ "사내망에 들어옴 = 벤더 절대 못 봄"은 동치가 아니다.
→ 임원 인사 결정, 노조 관련, 징계, 신원특정 가능 평가 코멘트 같은 것은 사외 LLM에 못 보낸다.

## 2. GAUSS vs 사외 LLM — Trade-off

### 2.1 GAUSS (사내 LLM)

- 오픈소스 모델 파인튜닝 기반 (삼성 Gauss)
- 모델 가중치 + 추론 + 로그 사내 통제
- vendor-side 로그 안 남기는 운영 가능
- 데이터 주권 100%, 벤더 의존도 0
- **약점**: 사외 프론티어 대비 성능 약함 (특히 복잡 추론, 긴 컨텍스트, 멀티모달)

### 2.2 사외 LLM

- 프론티어급 성능
- 일반 거버넌스 동반
- **약점**: 위 5가지 누수 채널 + 벤더 관할권 (외국 법 적용 가능)

### 2.3 라우팅 매트릭스 — 1축 아니라 2축

```
                    작업 복잡도 →
                    Low          Medium         High
민감도  Low      [GAUSS OK]    [GAUSS OK]    [사외 LLM]
   ↓    Med      [GAUSS]       [정책 ★]      [정책 ★]
        High     [GAUSS]       [GAUSS]        [GAUSS + 성능 손해]
```

**별표(★) 영역이 People팀의 자리.**
- 그대로 사외 LLM으로 → 보안 위험
- GAUSS로 → 답 품질 손실
- **마스킹/익명화 후 사외 LLM** → 정책으로 가능
- **요약/추상화 후 사외 LLM** → 정책으로 가능
- **분리 호출 (일부만 사외 LLM)** → 정책으로 가능

도메인 지식 없으면 결정 못 함. AX팀, 벤더, 일반 AI Gateway 모두 못 함.

### 2.4 Audit 책임 분담

| | vendor-side 로그 | application-side 로그 |
|---|---|---|
| GAUSS | LLM 자체 로그 미저장 (보안 강점) | **우리 플랫폼이 잡음** |
| 사외 LLM | 일부 남길 수도 | **우리 플랫폼이 잡음** |

→ "GAUSS 로그 미저장 + 우리 플랫폼 audit"가 기밀 영역에서 보안+감사 동시 충족의 조합.

### 2.5 분류 단계의 사내 완결 원칙

데이터 민감도 분류 단계에는 원문 HR 데이터가 들어온다. 이 입력 자체가 기밀 또는 민감 데이터일 수 있으므로 분류에는 사외 LLM을 절대 쓰지 않는다.

```
데이터 민감도 분류 Agent
 ├─ 조직×도메인 권한 단위 기본 등급 룩업
 ├─ PII 패턴 매칭
 ├─ 사내 LLM(GAUSS) 한국어 맥락 보강
 └─ 출력: {등급, 차폐 후보 항목, 신뢰도}
```

사외 LLM은 분류 결과와 차폐된 입력만 보고 처리한다. 원문 HR Data/Knowledge는 전사 공통 AI Platform에 적재하지 않는다.

## 3. AX팀 vs People팀 영역 분리

### 3.1 영역 정의 (정치적으로 가장 중요)

| 영역 | 주체 |
|---|---|
| 사외 LLM 도입 / 운영 | AX팀 |
| 사내 LLM(GAUSS) 인프라 | AX팀 또는 별도 인프라팀 (확인 필요) |
| 일반 AI Gateway / 라우팅 인프라 | AX팀 |
| **HR 도메인 민감도 분류 정책** | **People팀 (우리)** |
| **HR 도메인 워크플로우 / 승인 라인** | **People팀 (우리)** |
| **HR 데이터 거버넌스 정책** | **People팀 (우리)** |
| **원문 HR Data/Knowledge 적재·권한·자산화** | **HR AX Platform (우리)** |

### 3.2 충돌 회피 프레임

데모/포지셔닝 시 다음 프레임 일관 유지:
- ❌ "사외 LLM을 대체합니다"
- ⭕ "AX팀이 깐 인프라 위에 People팀 도메인 정책이 얹힙니다"
- ⭕ "AI Tool이 많아질수록 우리 플랫폼도 좋아집니다 (BYOM)"

**AX팀 영역 침범 인상 절대 금지.** AI Gateway 인프라를 우리가 build하면 충돌. 우리는 HR 원문 데이터와 Knowledge를 독립 통제하고, AX팀의 AI Tool은 허용된 방식으로 연계한다.

### 3.3 People팀 내부 호재

- **정보보호센터가 People팀 소속** — 강제 정책 합의에 결정적
- People팀장이 "하라"고 하면 부서 차원 협력 가능
- AXN 컬쳐이노베이션 랩이 People팀장에게 정책 제안 올릴 수 있는 위치

## 4. 데이터 저장 위치

데이터 저장 위치는 두 군데로 단순화한다.

| 위치 | 역할 |
|---|---|
| **DW** | 원천 시스템 데이터를 모은 통. GHRP(W), GHRP(S), 삼성U, 채용Hub 등 원천 데이터를 통합. |
| **HR AX Platform** | 운영 데이터, 표준 데이터, 자산 데이터를 적재. 권한·분류·자산화·RAG의 실행 위치. |

전사 공통 AI Platform에는 원문 HR Data/Knowledge를 적재하지 않는다. 사외 LLM이나 AX팀 AI Tool에는 HR AX가 권한 확인과 차폐를 거친 컨텍스트만 전달한다.
