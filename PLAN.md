## 2. 정치 트랙 (병렬, 우선)

### 2.1 P0 — 5~6월

- [ ] **강제 정책 초안 v0 작성**
  - 범위: HR 데이터 처리 시 우리 플랫폼 통과 필수. 단순 일반 질의는 제외.
  - "단순 사용 vs 민감 사용" 경계는 분류표 v0 참조.
  - 첫 협의 자료에 보안/컴플라이언스 부서가 좋아할 프레이밍: "엔터프라이즈 직접 사용 시 발생할 데이터 누수 위험을 HR 플랫폼이 사전 분류로 차단".

- [ ] **민감도 분류표 v0 작성**
  - HR 도메인 데이터 카테고리별 민감도 등급 (Low / Med / High)
  - People팀 + 정보보호센터 + 노무 일치 필요
  - 처음부터 완벽 못 만든다. v0는 좁게 시작 → 운영하며 확장.

- [ ] **People팀장 컨센서스 확보**
  - v3 가설 1페이지 요약 + 7월 도래 시 위험 시나리오 + 강제 정책 정당성 자료
  - "왜 People팀이 이걸 해야 하는가"의 답을 People팀장이 외부에 말할 수 있게.

### 2.2 P1 — 6~7월

- [ ] **정보보호센터 동의** — 같은 People팀 내라 정치적으로 가능. 강제 정책의 보안 정당성 확보.
- [ ] **AX팀과 영역 분리 합의** — "AX팀 인프라 위에 우리 도메인 정책" 프레임. AI Gateway는 AX팀 영역으로 인정.
- [ ] **노무팀 사전 협의** — "감시 도구"로 비치지 않게 자산 누적의 동의/투명성 메커니즘 설명.
- [ ] **법무 사전 검토** — HR 데이터 처리 강제의 법적 근거 정리.

### 2.3 P2 — 7월 이후

- [ ] **강제 정책 발효** — 엔터프라이즈 도래 직후 즉시 발효. 늦으면 사용자 우회 학습.
- [ ] **CISO/보안위원회 공식 승인** — 강제 정책 공식 문서화.

## 3. 코드 트랙

### 3.1 P0 — 5~7월

- [ ] **(가) 분류표 v0 데이터화**
  - 정치 트랙의 분류표를 코드에서 사용 가능한 구조로 (yaml/json + loader)
  - 카테고리, 등급, 패턴 정의

- [ ] **(나) 라우팅 엔진 v0**
  - 입력: 사용자 요청 + Work Card 컨텍스트 + 분류 결과
  - 출력: 모델 선택 (Gauss / 엔터프라이즈 / 마스킹 후 엔터프라이즈), 변환 지시
  - BYOM 인터페이스로 추상화 — 모델 슬롯 교체 가능

- [ ] **(다) 마스킹/익명화 변환기**
  - 별표 셀(중간 민감 + 복잡 추론)에서 사용
  - 임직원 식별자 → 익명 ID, 부서명 → 일반화, 금액 → 범위 등
  - 변환 정책도 People팀 합의 자산

- [ ] **(라) Application-side Audit Log**
  - 모든 LLM 호출의 정책 결정 / 입력 / 출력 / 근거 기록
  - Gauss든 엔터프라이즈든 동일하게
  - 이건 7월 보안 합의의 핵심 근거

### 3.2 P1 — 7~9월 (Accumulation 파이프라인)

`AXIOM/Agentwork.md`의 데이터 모델 차용. Phase 0~4 기준:

- [ ] **Phase 0/1: Work Card 도입**
  - `work_cards` 테이블 + 세션→카드 연결
  - 모든 작업이 카드 위에서 시작되게

- [ ] **Phase 2: Tacit Candidate 추출**
  - 대화 중 intent signal → `tacit_candidates`
  - assistant marker + 서버 측 추출 두 경로

- [ ] **Phase 3: Context Tag**
  - audience / sensitivity / exception 등 최소 태그 세트
  - retrieval의 eligibility filter

- [ ] **Phase 4: Review Queue**
  - reviewer decision flow (fact_rejected / context_split / approved_team / approved_standard / merged)
  - 충돌 탐지

### 3.3 P2 — 9~11월 (출시 준비)

- [ ] **Phase 5: Trust Tier Promotion**
  - team asset / standard asset 분리
  - Agent는 reviewed 이상만 사용

- [ ] **Workflow Orchestration v1**
  - L4/L5/L6 자산 위 다단계 흐름
  - 결재/승인 라인 시각화

- [ ] **UX — "강제가 부드럽게" 만들기**
  - 콜드 스타트가 채팅창이 아니라 "내 Work Card + 회사 컨텍스트"
  - 사용자가 Skill 단어 모른 채 일 끝낼 수 있게

- [ ] **데모 시나리오 3종 준비**
  - 시나리오 A: 임직원 정보 처리 → Gauss 라우팅 표시
  - 시나리오 B: 채용 트렌드 분석 → 엔터프라이즈 라우팅 표시
  - 시나리오 C: 평가 코멘트 분석 → 마스킹 후 엔터프라이즈 표시

## 4. 의사결정 필터

빌드 중 모든 기능에 다음 필터 적용:

1. **이 기능, Claude Enterprise / Gemini / Custom GPT로도 동일 가능?** → Yes면 위에 Mandate/Routing/Accumulation 중 하나가 얹혀야 함.
2. **이 기능, 사용자가 Skill/Agent/Tool 단어를 알아야 동작?** → Yes면 UX 재설계.
3. **이 기능이 자산 누적을 만드는가?** → No면 우선순위 낮춤.

## 6. 11월 출시물의 한 줄 정의 (목표)

> "L4/L5/L6 프로세스 자산 위에 깔린 HR Work Card 운영면. 사용자 작업이 들어오면 분류·라우팅·마스킹을 거쳐 적절한 LLM(Gauss 또는 엔터프라이즈)으로 보내고, 결과는 Tacit Candidate 추출 + Review Queue를 통해 회사 자산으로 누적된다. 모든 호출은 application-side audit으로 추적된다."

이게 11월 출시 후 People팀장이 외부에 한 줄로 설명할 문장. 이 문장이 작동하면 v3 성공.
