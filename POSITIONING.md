# Positioning v3

> v1 (자산 OS) → v2 (운영 레이어) → v2.1 (라우팅) → v3 (Mandate + Routing + Accumulation)
> 진화 과정은 [DECISIONS.md](./DECISIONS.md) 참조

---

## 1. 한 줄

**HR AX Platform은 HR 데이터가 AI Tool로 가기 전에 반드시 거치는 HR 특화 안전 게이트다. 원문 HR Data/Knowledge는 전사 공통 AI Platform에 올리지 않고, 플랫폼 안에서 권한 확인·민감도 분류·비식별화·라우팅을 처리하며, 사용할수록 People 조직의 지식이 자산으로 축적된다.**

## 2. 왜 HR 특화 Platform인가

사내 LLM(GAUSS), 사외 LLM(Chat GPT / Gemini 등), 전사 공통 AI Tool은 모델·기능으로 활용한다. 하지만 HR Data/Knowledge 자체는 공통 Platform이나 사외 LLM에 직접 적재하지 않는다. HR 데이터는 보안 등급, 조직 권한, 도메인 맥락, 검수 책임이 동시에 붙어야 하며, 이 네 가지를 HR 도메인 밖에서 일반화하면 누수나 오권한 사용이 발생한다.

| 축 | HR AX가 별도로 맡아야 하는 이유 |
|---|---|
| **보안** | AI Tool 앞단의 안전 게이트. 원문 HR Data/Knowledge는 공통 AI Platform에 올릴 수 없음. 분류·비식별화·라우팅·audit을 HR AX가 통제해야 함. |
| **축적** | HR 담당자는 AI 빌더가 아님. 사업부/도메인별 RAG와 로컬 DB를 사람이 관리하면 세션 단위로 초기화됨. Work Card 사용 흐름에서 자동 축적돼야 함. |
| **권한** | DW와 플랫폼 적재 데이터는 접근 권한이 있는 사람만 활용해야 함. 조직×도메인 권한 단위로 필요한 만큼만 검색·RAG·Tool 실행을 허용해야 함. |

## 3. 한 장 아키텍처

```
─────────────────────────────────────────────────────────────────
[사용자 = HR 담당자]
       │
       │ HR 데이터 처리 시 강제 진입 (Mandate)
       ▼
─────────────────────────────────────────────────────────────────
[운영 레이어 — People팀의 단일 책임 영역]
   ├─ Authorization & Audit  ─ 누가/언제/무엇을, application-side log
   ├─ 데이터 민감도 분류 Agent ─ 분류표 v0 + PII 패턴 + 사내 LLM
   ├─ Permission Policy       ─ 조직×도메인 권한 단위별 데이터 접근
   ├─ Routing Policy         ─ 민감도 × 복잡도 매트릭스 기반 분기
   ├─ Masking / Anonymization ─ 마스킹 후 사외 LLM 호출 가능 영역
   └─ Workflow Orchestration ─ L4/L5/L6 자산 위 다단계 흐름
─────────────────────────────────────────────────────────────────
[실행 레이어 — Agent / Skill / Tool]
   ├─ Agent  : L5/L6 단위 실행체 (Process Coaching 자산 활용)
   ├─ System Agent : 분류 / 차폐 검사 / 자산 추출
   ├─ Skill  : 실행 레이어 노하우 md (versioned + scoped + reviewed)
   └─ Tool   : 코드 동작 (임직원 조회, 계산기, 출장비 계산 등)
─────────────────────────────────────────────────────────────────
[AI Tool 연결]
   ├─ 사내 LLM (GAUSS)          ← 기밀 원문 / 분류 보강 / 사내 임베딩 / LLM 자체 로그 미저장
   ├─ 사외 LLM (Chat GPT / Gemini 등) ← 일반 / 비식별화 후 민감 / 복잡 추론
   └─ 사내 AI Tool              ← HR AX 권한·분류·차폐 정책 통과 후 제한 연계
   ※ 사외 LLM은 원문 HR Data/Knowledge 직접 접근 금지
─────────────────────────────────────────────────────────────────
       │
       ▼
[자산 누적 (Accumulation)]
   사용 → Tacit Candidate 추출 → Context Tagging →
   Review Queue → 자산 신뢰 단계 승급 → 다음 사용의 출발점
─────────────────────────────────────────────────────────────────
```

## 4. 3대 차별 축 상세

### 4.1 보안 — 안전 게이트

**원칙.** HR 담당자가 HR 데이터(임직원 정보, 평가, 보상, 인사위 등)를 처리하는 모든 행위는 우리 플랫폼을 통과해야 한다. 단순 일반 질의(번역, 문서 요약, 일반 정보 검색)는 제외.

**왜 강제가 필요한가.**

- 강제 없으면 사용자가 직접 사외 LLM으로 간다. 콜드 스타트 문제 영원히 못 푼다.
- 자유 사용 시 임직원 개인정보가 사외 LLM 컨텍스트에 그대로 올라간다. 보안 사고 시간 문제.
- 강제는 사용자 베이스 확보의 채찍 + 보안/CISO 동맹의 정당성을 동시에 만든다.
- 원문 HR Data와 HR Knowledge는 전사 공통 AI Platform에 적재하지 않는다.

**경계 정의.** 강제 범위는 분류표 v0가 정의한다. 강제 정책 발효 전에 분류표가 합의돼야 한다.

**부드러운 강제.** 채찍만으로는 우회를 부른다. UX가 충분히 좋아서 "어차피 더 편하다"가 돼야 진짜 작동한다. 첫 데모에서 이걸 증명해야 정책이 살아남는다.

### 4.2 보안 — 민감도 × 복잡도 라우팅

**1축이 아니라 2축.** 사내 LLM(GAUSS)은 데이터 주권이 강하고 LLM 자체 로그 미저장이 가능하지만 성능 제약이 있고, 사외 LLM은 성능이 강하지만 원문 HR Data/Knowledge를 직접 볼 수 없다. 두 축을 같이 봐야 권한 단위별 trade-off 결정이 가능하다.

**분류는 사내에서 완결.** 분류 단계 입력은 민감 데이터일 수 있으므로 사외 LLM을 절대 쓰지 않는다. 데이터 민감도 분류 Agent가 조직×도메인 기본 등급 룩업, PII 패턴 매칭, 사내 LLM(GAUSS) 한국어 맥락 보강을 통합 처리한다.

```
데이터 민감도 분류 Agent (사내 전담)
 ├─ 1차: 조직×도메인 기본 등급 룩업
 ├─ 2차: PII 패턴 매칭 (사번/이름/평가/징계 등)
 ├─ 3차: 사내 LLM 호출 (한국어 맥락 보강)
 └─ 출력: {등급, 차폐 후보 항목, 신뢰도}
```

```
                    작업 복잡도 →
                    Low          Medium         High
민감도  Low      [GAUSS OK]    [GAUSS OK]    [사외 LLM]
   ↓    Med      [GAUSS]       [정책 필요]    [정책 필요 ★]
        High     [GAUSS]       [GAUSS]        [GAUSS + 성능 손해]
```

**별표(★) 영역이 People팀 정책의 핵심 자리.** "중간 민감 + 복잡 추론" 영역에서:

- 그대로 사외 LLM으로? → 보안 위험
- 사내 LLM(GAUSS)로? → 답 품질 손실
- **마스킹/익명화 후 사외 LLM으로** → 정책으로 가능
- **요약/추상화해서 사외 LLM으로** → 정책으로 가능
- **분리 호출 (일부만 사외 LLM으로)** → 정책으로 가능

이 영역의 결정은 도메인 지식 없으면 못 한다. AX팀도, 벤더도, 일반 AI Gateway도 못 한다. **People팀만 가능.**

### 4.3 축적 — 자산 누적 파이프라인

**핵심 통찰** ([AXIOM/Agentwork.md](../AXIOM/Agentwork.md) 인용):
> "HR AX의 moat는 대화가 많아서가 아니다. 문서가 많아서가 아니다. **업무 카드 위에서 포착된 판단 기준이 검토를 거쳐 재사용 자산으로 승격되는 구조**에서 생긴다."

HR 담당자는 대부분 Light user다. 사업부/도메인별 RAG를 직접 구축하거나 로컬 DB를 철저히 관리하는 방식은 확산되기 어렵다. 그래서 Work Card 사용 흐름 안에서 표준 데이터, 자산 데이터, Skill이 자동으로 붙고, 사용 흔적이 검수 큐로 넘어가야 한다.

**파이프라인:**

```
사용자 작업 (Work Card)
    ↓ Copilot과 대화 진행
    ↓ Tacit Candidate 추출 (서버 측 intent capture)
    ↓ Context Tag 부착 (audience, sensitivity, exception 등)
    ↓ 작성자 공개 범위 제안 (디폴트: 조직×도메인 권한 단위 한정)
    ↓ Review Queue 진입
    ↓ Reviewer 검수
    │   └─ fact_rejected / context_split / approved_team
    │      / approved_standard / merged
    ↓ 자산 신뢰 단계 승급
    ↓ 다음 Work Card의 검색 대상 / Agent 활용 자산
    ↓ 반복 패턴은 Lab이 신규 L6 / 신규 Skill / 둘 다 / 둘 다 아님으로 결정
```

**자산 신뢰 단계:** raw signal → tacit candidate → 검수 통과 자산 → 권한 단위 자산 → 전사 표준 자산.
Agent는 최소 검수 통과 자산 이상만 사용한다.

**핵심 비목표:** raw chat 전체를 자동 승격하지 않는다. 모든 신호가 자산이 되지 않는다. 사람의 검수가 들어가는 게 핵심.

**두 가지 승격 경로.** ③ 자산 데이터가 규정/기준/업무 프로세스로 정형화될 때만 ② 표준 데이터로 승격한다. LLM 운영 노하우로 추출 가능하면 Skill로 승격한다. 다수의 자산은 ③ 자산 데이터에 잔류한다.

### 4.4 권한 — 조직×도메인 데이터 접근

데이터는 두 군데에 저장된다. DW는 원천 시스템 데이터를 모은 통이고, HR AX Platform은 운영 데이터·표준 데이터·자산 데이터를 적재한다. 모든 검색, RAG, Tool 실행은 사용자 권한 단위 안에서만 일어난다.

```
[원천 시스템] → [DW: 원천 데이터 통합 저장소]
                         │
                         ▼
              [HR AX Platform]
                 ├─ ① 운영 데이터
                 ├─ ② 표준 데이터
                 └─ ③ 자산 데이터

[사용자] → [조직 역할 + 담당 도메인] → [권한 단위]
                                      ├─ DW 접근 가능 범위
                                      ├─ 표준 데이터 접근 가능 범위
                                      └─ 자산 데이터 공개 범위
```

## 5. 데모 시나리오 3종 (11월 출시 기준)

데모는 한 화면으로 끝까지 보여준다. 사용자가 입력 → 라우팅 결정 시각화 → 결과 → 자산 누적 한 줄 표시.

### 시나리오 A — 민감 데이터 → 사내 LLM(GAUSS)

**사용자 입력**: "다음달 임원 인사위 안건 정리해줘. 작년 임원 A씨 평가 이력 같이 봐줘."

**플랫폼 동작:**
- 분류기 결과: `sensitivity=High` (임원 식별자 + 평가 이력)
- 라우팅 결정: **사내 LLM(GAUSS)**
- 화면 표시: "임원 인사 데이터 감지 → 사내 LLM(GAUSS)로 처리합니다 (사외 LLM 차단, 로그 미저장)"
- GAUSS 응답
- Audit log: `model=gauss, sensitivity=high, reason=executive_personnel_data`
- 자산 후보: "임원 인사위 안건 정리 시 작년 평가 이력 인용 패턴" — Review Queue 진입

### 시나리오 B — 비민감 + 복잡 → 사외 LLM

**사용자 입력**: "올해 IT 업계 채용시장 트렌드 정리해줘. 우리 직무군 중심으로."

**플랫폼 동작:**
- 분류기 결과: `sensitivity=Low, complexity=High`
- 라우팅 결정: **사외 LLM**
- 화면 표시: "공개 정보 + 복잡 분석 → 사외 LLM으로 처리합니다"
- 사외 LLM 응답
- Audit log: `model=external_llm, sensitivity=low, reason=public_market_analysis`
- 자산 후보: 자료 인용 / 직무군 매핑 — Review Queue

### 시나리오 C — 별표 영역 → 마스킹 후 사외 LLM

**사용자 입력**: "임원 A씨 평가 코멘트 기반으로 강점·약점 분석해서 1on1 코칭 시나리오 만들어줘."

**플랫폼 동작:**
- 분류기 결과: `sensitivity=Med (식별 가능), complexity=High (코칭 시나리오 작성)`
- 라우팅 결정: **마스킹 후 사외 LLM**
- 화면 표시 (좌): 원본 코멘트 ("임원 A씨", "OO부서", "30억 매출")
- 화면 표시 (우): 마스킹된 입력 ("임원_X", "사업부_A", "[금액 범위 1]")
- 사외 LLM 응답 (마스킹 입력 기반)
- 후처리: 응답 내 placeholder를 원본으로 역치환
- Audit log: `model=external_llm, sensitivity=med, masked=true, masking_policy=v0.3`
- 자산 후보: "임원 코칭 시나리오 작성 패턴", "마스킹 정책 적용 사례" — Review Queue

이 세 시나리오 한 화면에 보여주면 People팀장 설득 가능하다. **"사외 LLM은 빈 화면, 우리는 회사 맥락 + 안전 게이트 + 시간이 갈수록 좋아짐"**.

## 6. 차별 후보 5가지 평가

| # | 후보 | 강도 | 근거 |
|---|---|---|---|
| 1 | 민감도 기반 LLM 라우팅 | 강 | HR 도메인 분류표와 분류 Agent 정책은 People팀만 정의 가능. AI Gateway 인프라는 AX팀 영역, 정책은 우리 영역. |
| 2 | 다단계 워크플로우 오케스트레이션 | 강 | 사외 LLM은 단발 챗에 강함, 다단계 묶음에 약함. L4/L5/L6 자산 위에서만 가능. |
| 3 | HR 도메인 권한·승인 정책 | 강 | 조직×도메인 권한 단위가 DW·표준 데이터·자산 데이터 접근을 동시에 통제. |
| 4 | 사내 HR 시스템 연동 | 중 | 인사정보/평가 등 HR 원천 데이터는 DW를 통해 필요한 범위만 활용. |
| 5 | HR 도메인 특화 UX | 약 | 단독 moat 아님. 1~4와 결합 시 가시화. |

**1, 2, 3이 본체. 4, 5는 부가.**

## 7. 절대 잊지 말 것

### 7.1 의사결정 필터

빌드 중인 모든 기능에 다음 질문 적용:

> "이 기능, 사외 LLM(Chat GPT / Gemini 등)이나 Custom GPT로도 동일하게 가능한가?"

Yes면 차별 못 만든다. 위에 보안/축적/권한 중 하나가 얹혀야 차별이 생긴다.

### 7.2 Mandate의 정당성 라인을 흔들지 말 것

"Skill이 모여 있어서 편함"은 강제 정당성으로 약하다. 정당성은 다음 셋에서만 나온다:

- HR 데이터 거버넌스 (외부 누수 위험 차단)
- 회사 맥락 누적 (회사 자산화)
- HR 도메인 권한 체계의 단일 책임 부서

이 셋을 People팀장, 정보보호센터, 보안, AX팀에 일관되게 강조해야 정책 합의 가능.

### 7.3 자산 누적이 가장 큰 moat

[AXIOM/Agentwork.md](../AXIOM/Agentwork.md)의 Work Card / Tacit Candidate / Context Tag / Review Queue / 자산 신뢰 단계 데이터 모델이 v3 Accumulation과 정확히 일치한다. **이 설계를 차용해서 11월 빌드 P0의 일부로 가져간다.**

사외 LLM이 단독으로 카피 못 하는 본질이 여기 있다. 시간이 갈수록 강해지는 메커니즘.
