## Quick Read

**한 줄 정의**

> HR AX Platform은 **HR 담당자가 상용 LLM을 안전하게 쓰는 관문**이자, **쓸수록 People 조직의 자산이 누적되는 Workbench**.

**3개 핵심 메커니즘**

01. 상용 LLM을 사용하기 위한 **안전한 관문** (데이터 등급 자동 분류 + 비식별화 게이트 + 사내 재매핑)
02. **Work Card 기반 업무 실행 로그** (정형/비정형 모두 1급 시민)
03. **검수형 자산화** (반복 패턴 → L6 후보 → Domain Experts 검수 → 정형 승급)

**청중별 한 줄**

| 청중 | 메시지 |
|---|---|
| 부문 People팀장·임원 | "민감 정보가 사외로 나가지 않습니다 + People 조직 지식 자본화" |
| 정보보호센터 | "비식별화 게이트 + 매핑은 사내 보관 + 응답 수신 시 사내 재매핑" |
| HR 담당자 | "안전하게 외부 LLM 활용 + 쓸수록 다음 일이 쉬워짐" |
| AX팀 | "응답 모델은 외부 상용 LLM. 인프라 영역 비침범" |

# §01 왜 재정의했는가

**Section Label**: Why Now
**Headline**: 상용 LLM이 사내망 안으로 들어오는 순간, 차별의 정의가 바뀐다

## 1.1 전제

- 사내망에서 **상용 LLM(Claude / Gemini / KODEX 등) 직접 사용 가능** 환경 도래
- 사용 주체는 People팀 + 다른 부서 + 임직원 일반
- 이전 가설("우리만 외부 LLM 못 씀")은 **무효화**

## 1.2 기존 v1 가설의 약점

- **"L4/L5/L6 데이터가 moat"**: 같은 데이터를 엔터프라이즈가 사내망 안에서 그대로 접근
- **"채팅 UI"**: 엔터프라이즈가 동일 UX 제공
- **"Skill 라이브러리"**: 한 명의 노력으로는 엔터프라이즈 산하 ecosystem 따라잡기 불가

→ 데이터·UI·라이브러리 어느 것도 단독으로는 차별이 안 됨

## 1.3 v3가 잡은 새 차별 축

- **People팀만의 강제력** (부문 People팀장 선언) → 진입의 정당성
- **HR 도메인 한정 비식별화 게이트** (셀별 차폐 정책 = 도메인 지식)
- **People Knowledge 누적** (사용 흔적이 People 조직 자산으로)

→ 셋 다 엔터프라이즈가 단독으로 못 만드는 것

# §02 두 축 — 안전한 관문 × 자산화

**Section Label**: Two Axes
**Headline**: 강제 진입의 정당성과 시간이 우리 편인 메커니즘

## 2.1 축 1 — 상용 LLM을 사용하기 위한 안전한 관문

**데이터 등급 3단 분류 → 처리 경로 분기**

| 등급 | 처리 | 모델 |
|---|---|---|
| 기밀 | 사외 송출 금지 | Gauss (사내, 로그 없음) |
| 민감 | 비식별화 게이트 후 외부 | 상용 LLM |
| 일반 | 그대로 외부 | 상용 LLM |

**자동 분류 3계층**: 셀(조직×도메인) 기본 등급 → 키워드/패턴 PII detection → 사용자 동의 (상향만 가능, 하향 불가)

```
[사용자 입력] ──▶ 등급 자동 분류
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
       기밀           민감            일반
        │              │              │
       Gauss      비식별화 게이트   그대로 송출
        │              │              │
        │         매핑 사내 보관       │
        │              │              │
        │         사내 재매핑          │
        │              │              │
        ▼              ▼              ▼
                  [사용자 출력]
```

- **메시지**: "외부에 직원 정보가 나가지 않습니다"
- **차폐 정책표**: (조직 × 도메인) 셀별 무엇을 차폐할지 다름 → 도메인 지식
- **Function Calling 패턴**: 외부 LLM은 Tool/Skill/Agent 직접 접근 X. 의도만 표현, 실행은 플랫폼 내부
- **강제 정당성**: HR 데이터 특성상 직접 외부 LLM 사용 시 인사비/개인정보 누설 우려

## 2.2 축 2 — 사용 흔적의 People Knowledge化

- **메시지**: "쓸수록 People 조직의 자산이 됩니다"
- **메커니즘**: Work Card → 사용 누적 → 검수형 Asset Registry 등록
- **승급 단계**: 개인 → 셀 → 도메인 종방향(부문) → 전사 표준
- **임원 라벨**: "People 조직 지식 자본화"

## 2.3 두 축이 결합돼야 하는 이유

| 단독 사용 시 약점 | 결합 시 |
|---|---|
| 안전한 관문만 → "DLP 솔루션 사면 되잖아" 천장 낮음 | 강제 진입의 정당성 + 시간이 우리 편 |
| 자산화만 → "왜 강제로 써야 함?" 정당성 부족 | 진입 → 안전 사용 → 흔적 자산화 → 다음 사용 가치↑ → 자발적 사용 ↑ |

# §03 조직 × 도메인 매트릭스

**Section Label**: Workbench Coordinate
**Headline**: 모든 권한·민감도·자산 공개 범위는 (조직 × 도메인) 셀에서 결정

## 3.1 X축 — 조직 (열람 범위 계층)

| 레벨 | 주체 | 열람 범위 |
|---|---|---|
| L0 Admin | AX&CI Lab | 전사 모든 셀 |
| L1 부문(본사) | 부문 People팀 | 자기 부문 + 산하 모든 사업부 |
| L2 사업부 | 사업부 담당자 | 자기 사업부만 |

**사업부 8개**: VD, DA, MX, NW, 의료기기, SR, 생기연, CDO

## 3.2 Y축 — 도메인 (12개)

교육 / 노사 / 보상·근태 / 인력운영 / 임원조직 / 제도 / 채용 / 총무 / 해외인사 / DEI / 조직문화 / 집단지성

## 3.3 셀별 라벨링 (분류표 v0)

각 셀에 4가지 라벨이 부착됨:

01. **데이터 접근 권한** — 어느 인사 DB 영역까지 열람 가능
02. **기본 민감도** — 부문이라고 모두 High 아님. 셀별 개별 라벨링 필요
03. **차폐 정책** — 어느 항목을 어떻게 비식별화할지
04. **자산 승급 검수자** — Domain Experts 지정

**셀 수**: 9 조직 × 12 도메인 ≈ ~96 셀 (도메인별 사업부 전개 여부에 따라 가감)

## 3.4 자산 공개 범위 (5단계)

| 범위 | 정의 | retrieval 가능자 |
|---|---|---|
| 개인 | 작성자 본인 | 본인만 |
| 셀 한정 | (조직, 도메인) 셀 | 같은 셀 부착자 |
| 도메인 종방향 | 같은 도메인, 조직 계층 따라 | 부문은 산하 사업부 셀 자산 retrieval |
| 조직 횡방향 | 같은 조직, 모든 도메인 | 1차 출시 미지원 |
| 전사 표준 | 모든 셀 | 누구나 |

# §04 인사 DB 아키텍처

**Section Label**: Data Architecture
**Headline**: 원천 시스템 → DW → HR AX Platform 자체 적재 3종

## 4.1 원천 시스템

| 시스템 | 담는 것 | 비고 |
|---|---|---|
| GHRP(S) | 급여, 근태, 평가 | GHRP(W) 토대 위에 구성 |
| GHRP(W) | 조직, 개인, 발령, 직무 | 마스터 |
| 기타 | 삼성U, 채용 Hub 등 | |

## 4.2 DW (Digital Warehouse)

- 위 3종 원천 통합 저장소
- 업데이트 주기: **1일 1회**
- HR AX Platform은 **DW에서만** 인사 데이터 가져옴 (원천 직접 X)
- 단 추후 GHRP(W)/(S)는 **API로 직접 연계 예정** (실시간 케이스)

## 4.3 HR AX Platform 자체 적재 3종

### ① 운영 데이터 (Master)
시스템이 갖춰야 할 최소 데이터. 로그인·권한 관리를 위한 사용자 식별 정보, 조직·개인·발령 등 플랫폼 자체 기능 수행에 필수적인 마스터 데이터. DW가 아닌 연계용 DB에서 1시간 단위로 동기화.

### ② 표준 데이터 (Standards)
부서·전사 차원에서 공식적으로 운영되는 자료. 인사 제도, 규정, 업무 프로세스(L4/L5/L6), 표준 양식 등. AX&CI Lab이 운영하며 잦은 수정·생성을 직접 관리. 비정형 자료를 LLM이 이해하도록 각주·설명을 첨부하여 RAG·벡터화·학습에 사용.

### ③ 자산 데이터 (Knowledge)
HR 담당자의 사용 흔적이 누적되어 형성되는 People 조직 자산. Workbench의 Work Card 사용에서 발생한 노하우·판단·결정이 Tacit Candidate로 큐잉되고, Domain Experts 검수를 거쳐 자산 데이터에 등재. 셀(조직×도메인) 좌표와 공개 범위(개인 / 셀 / 도메인 종방향 / 전사 표준)를 부착.

| # | 명칭 | 출처 | 주기 | 운영 |
|---|---|---|---|---|
| ① | 운영 데이터 (Master) | 연계용 DB | 1시간 | 시스템 자동 |
| ② | 표준 데이터 (Standards) | 직접 관리 | 잦은 수정 | AX&CI Lab |
| ③ | 자산 데이터 (Knowledge) | 사용 누적 | 실시간 | Domain Experts 검수 |

## 4.4 그림

```
[원천]
  GHRP(W) ─→ GHRP(S)        삼성U, 채용Hub
       │         │                 │
       └─────────┴─────────────────┘
                 ▼
                DW (1일 1회, read-only)
                 │  ┌───── (예외) GHRP API 직접 연계
                 ▼  ▼
       ─────────────────────────────────
       HR AX Platform
       ─────────────────────────────────
       ① 운영 데이터 (Master)    — 연계용 DB, 1시간
       ② 표준 데이터 (Standards)  — Lab 운영, 비정형
       ③ 자산 데이터 (Knowledge)  — 검수 통과 자산
                 │
                 │ 반복 패턴 검수 통과 시
                 ▼
       ② 승격 (③ 자산 데이터 → ② 표준 데이터)
```

# §05 자산 승급 파이프라인

**Section Label**: Promotion Pipeline
**Headline**: 비정형 → 정형 → 인사제도/규정 승격은 모두 사람 검수를 거친다

## 5.1 단계

```
Step 1. Work Card 작성 / 종료
   └─ 사용 흔적이 자유 텍스트로 누적

Step 2. Tacit Candidate 자동 큐잉
   └─ 카드 종료 시 "결정적 판단 한 줄" 입력
   └─ 시스템이 결정 지점·반복 패턴 자동 감지

Step 3. Context Tag 부착
   └─ 셀(조직×도메인), audience, processStep, bucketScope
   └─ 본문은 비정형 그대로

Step 4. Domain Experts 검수 (승인 / 반려 / 보류)
   └─ 승인: ③ 자산 데이터에 등재. 작성자·검수자 양쪽 실적
   └─ 반려: 작성자 피드백. 일정 기간 후 자동 폐기 또는 수정 후 재제출
   └─ 보류: 추가 정보 요청. 30일 미보강 시 자동 폐기

Step 5. ③ 자산 데이터에 등재
   └─ 공개 범위 (개인/셀/도메인종방향/전사) 부착
   └─ 다른 카드에서 retrieval 가능

Step 6 (선택). ② 표준 데이터 승격
   └─ 전사 표준 자산 + AX&CI Lab 추가 검수
   └─ ② 표준 데이터로 편입
```

## 5.2 정형 승급 — 비정형이 L6가 되는 흐름

- L4/L5/L6에 정의된 업무 = **정형 업무**
- 그 외 = **비정형 업무** (실제 People팀 일감의 다수)
- **반복 패턴이 신규 L6 후보로 자동 포착되고, Domain Experts 검수를 거쳐 정형 프로세스로 승격됨**
- 승급된 L6는 Process Coaching AI 트리에 추가 → 다음 사람한테는 정형 업무로 매칭

## 5.3 부서원 작업물 차용 (Agent / Skill / Tool)

| 차용 항목 | 채택 | 노출 방식 |
|---|---|---|
| Agent / Skill / Tool 정의 | ⭕ | 실행 레이어 용어로 사용 |
| Skill 라이브러리 사용자 직접 선택 | ❌ | Work Card에 시스템 자동 부착 |
| 콜드 스타트 = 빈 채팅창 | ❌ | Work Card 보드가 콜드 스타트 |

**핵심**: "Skill 기반"이 아니라 "**Work Card에 Skill·Tool 자동 결합**"

# §06 우려와 정찰 항목

**Section Label**: Risks & Recon
**Headline**: 자산화는 자동으로 굴러가지 않는다 — 4가지 조건이 모두 갖춰져야

## 6.1 자산화 실패 모드

| 실패 모드 | 원인 | 대응 |
|---|---|---|
| 사용자가 결과만 받고 끝 | 노하우 추출 트리거 부재 | Card 종료 시 1줄 입력 강제, 결정 지점 자동 감지 |
| Tacit Candidate 큐 폭증 | 검수자 부담 | 자동 1차 분류, 유사 자산 머지, 검수자 적재 한도 |
| Domain Expert 검수 안 함 | 인센티브 없음 | People팀 차원 인정·평가 연계 |
| 자산이 retrieval에서 안 뜸 | 임베딩/태그 품질 | 한국어 임베딩 검증, 거짓 양성 모니터 |
| 자산이 떠도 사용자가 무시 | 신뢰 미형성 | 출시 초기 셀 1~2개로 작동 검증 후 확대 |

## 6.2 정찰 항목 (P0)

01. **부문 People팀장 입장** — 강제 사용 선언이 가능한가
02. **분류표 v0 작성 가능 인력** — 셀×도메인 기본 등급(기밀/민감/일반) 라벨링 합의 누가
03. **Domain Experts 지명 권한** — 셀별 검수자 누가 정하는가
04. **부서원 가설 통합 가능 여부** — Skill 정의 차용에 대한 합의

## 6.3 정찰 항목 (P1)

01. **GHRP(W)/(S) 직접 API 연계 가능 시점** — DW 1일 1회 한계 보완
02. **사내 임베딩 모델 가용성** — 자산 retrieval용 (Gauss 등)
03. **정보보호센터 협조 수준** — 매핑 사내 보관 정책 합의
04. **AX팀 게이트웨이 정책** — 외부 LLM 직접 접근 시 분류·로깅 방식

## 6.4 1차 출시 시점 정의

> "**부문 People팀 1개 + 사업부 People팀 1개에서 (조직×도메인) 셀 2~3개를 대상으로, 등급 분류 + 비식별화 게이트 + Work Card + 검수형 자산화가 작동하는 v1**"

- 광범위 자산화는 무리
- 본인 소속 셀 / 부문 People팀 셀 / 임원조직 셀 우선
- 자산 30~100건, Domain Expert 검수자 5~10명 규모

# Appendix A — 용어 정리

| 용어 | 정의 |
|---|---|
| Work Card | 사용자 일감 단위. 채팅 세션 아님. 정형/비정형 모두 1급 시민 |
| Workbench | 사용자가 보는 UI 공간. (조직×도메인) 셀 단위로 분리 |
| 운영 데이터 (Master) | 데이터 종 ①. 로그인·권한·마스터 |
| 표준 데이터 (Standards) | 데이터 종 ②. AX&CI Lab 운영. 제도·규정·L4/L5/L6 |
| 자산 데이터 (Knowledge) | 데이터 종 ③. 검수 통과한 People 조직 자산 |
| 데이터 등급 | 기밀 / 민감 / 일반. 처리 경로 결정 |
| 비식별화 게이트 | 민감 등급 사외 송출 시 차폐, 응답 수신 시 사내 재매핑하는 단일 통제점 |
| Function Calling | 외부 LLM이 Tool 호출 의도만 표현, 실제 실행은 플랫폼 내부에서 수행하는 패턴 |
| 분류표 v0 | (조직×도메인) 셀별 데이터 접근 권한·기본 등급·차폐 정책·Domain Experts 라벨링 표 |
| Domain Experts | 셀별 자산 검수자. 부문/사업부에서 선별된 담당자 |
| Tacit Candidate | 검수 전 후보 자산. 자유 텍스트 본문 |
| Trust Tier | 자산 신뢰 등급 (raw → reviewed → team → standard) |
| BYOM 슬롯 | Bring Your Own Model. 응답 모델 인터페이스. 외부 상용 LLM 또는 Gauss 꽂힘 |

# Appendix B — 청중별 한 페이지

## B.1 부문 People팀장·임원

> "**HR 담당자가 외부 LLM을 안전하게 쓰는 도구이자, 사용 흔적이 People 조직 자산으로 누적되는 Workbench**입니다.
> 민감 정보는 사외에 나가지 않습니다.
> 부문 People팀 차원의 강제 사용 선언으로 진입을 잡고,
> 시간이 갈수록 People 조직의 지식 자본이 됩니다."

## B.2 정보보호센터

> "**비식별화 게이트가 단일 통제점**입니다.
> 매핑 테이블은 사내에만 보관되며, 외부에는 차폐된 업무 맥락만 전달됩니다.
> 응답 수신 시 사내 재매핑으로 복원합니다.
> Application-side 감사 로그가 모든 호출을 기록합니다."

## B.3 AX팀

> "응답 모델은 등급별로 분기됩니다 — 기밀은 사내 Gauss, 민감/일반은 외부 상용 LLM.
> BYOM 슬롯으로 모델 교체 가능합니다.
> 외부 LLM과는 Function Calling 패턴으로만 연결되며, Tool 실행은 모두 플랫폼 내부.
> 인프라 영역은 침범하지 않습니다."

## B.4 HR 담당자(사용자)

> "**Work Card에 일감 단위로 작업합니다.**
> 민감 정보 처리는 자동으로 안전하게.
> 지난 카드의 판단·노하우가 다음 카드에 자동으로 따라옵니다.
> 매번 처음부터 시작하지 않습니다."
