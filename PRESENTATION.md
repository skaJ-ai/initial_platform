> HR AX Platform은 **원문 HR Data/Knowledge를 공통 AI Platform에 올리지 않고**, HR 데이터가 AI Tool로 가기 전에 권한 확인·등급 분류·비식별화를 수행하는 **HR 특화 안전 게이트**.

# §01 왜 구체화했는가

**Section Label**: Why Now
**Headline**: HR 데이터는 AI Tool 앞단에 별도 안전 게이트가 필요하다

## 1.1 전제

- 사내망에서 **사외 LLM(Chat GPT / Gemini 등)과 사내 AI Tool을 업무에 활용**하는 환경 도래
- 사용 주체는 People팀 + 다른 부서 + 임직원 일반
- 이전 가설("우리만 사외 LLM 못 씀")은 **무효화**

## 1.2 기존 소개안에서 더 선명해져야 할 지점

- **통합 창구**만으로는 부족: HR 원문 데이터가 AI Tool로 직접 흘러가는지 통제 설명 필요
- **Agent/Skill/Tool 관리**만으로는 부족: Skill은 데이터 종이 아니므로 권한·분류·차폐 체계와 분리 설명 필요
- **권한/자산화**는 기존 메시지지만 더 구체화 필요: DW, 운영/표준/자산 데이터, 조직×도메인 권한 단위, 검수 흐름을 그림으로 보여줘야 함

→ 기존 2장을 뒤엎는 것이 아니라, HR 데이터 특성상 필요한 안전 게이트와 권한 기반 자산화 구조를 명확히 얹는다.

## 1.3 v3가 강화한 축

- **보안**: 원문 HR Data/Knowledge는 전사 공통 AI Platform에 적재하지 않고, AI Tool 앞단에서 등급 분류·차폐·audit
- **축적**: Light user가 쓰기만 해도 Work Card 사용 흔적이 People 조직 자산으로 누적
- **권한**: 조직×도메인 권한 단위로 DW·표준 데이터·자산 데이터 접근 통제

→ 셋 다 HR 데이터 책임 부서가 직접 정의해야 하는 것

# §02 3대 축 — 보안 × 축적 × 권한

**Section Label**: Three Pillars
**Headline**: Platform 강화의 핵심은 안전 게이트, 축적, 권한이다

## 2.1 보안 — 사외 LLM을 사용하기 위한 안전한 관문

**데이터 등급 3단 분류 → 처리 경로 분기**

| 등급 | 처리 | 모델 |
|---|---|---|
| 기밀 | 사외 송출 금지 | 사내 LLM (GAUSS, 로그 미저장) |
| 민감 | 비식별화 게이트 후 외부 | 사외 LLM (Chat GPT / Gemini 등) |
| 일반 | 그대로 외부 | 사외 LLM |

**자동 분류 3계층**: 조직×도메인 기본 등급 → PII 패턴 매칭 → 데이터 민감도 분류 Agent(GAUSS 기반). 분류 입력은 민감 데이터일 수 있으므로 사외 LLM을 절대 호출하지 않음.

```
[사용자 입력] ──▶ 데이터 민감도 분류 Agent
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
       기밀           민감            일반
        │              │              │
       GAUSS      비식별화 게이트   그대로 송출
        │              │              │
        │         매핑 사내 보관       │
        │              │              │
        │         사내 재매핑          │
        │              │              │
        ▼              ▼              ▼
                  [사용자 출력]
```

- **메시지**: "외부에 직원 정보가 나가지 않습니다"
- **상향 원칙**: 사용자는 등급 상향만 가능. 즉시 적용 + audit log + 권한 단위 Domain Expert 통지 + 분류표 v0 갱신 트리거
- **차폐 정책표**: 조직×도메인 권한 단위별 무엇을 차폐할지 다름 → 도메인 지식
- **Function Calling 패턴**: 사외 LLM은 Tool/Skill/Agent 직접 접근 X. 의도만 표현, 실행은 플랫폼 내부
- **강제 정당성**: HR 데이터 특성상 직접 사외 LLM 사용 시 인사비/개인정보 누설 우려
- **원칙**: 원문 HR Data/Knowledge는 전사 공통 AI Platform에 올리지 않음

## 2.2 축적 — 사용 흔적의 People Knowledge化

- **메시지**: "쓸수록 People 조직의 자산이 됩니다"
- **전제**: HR 담당자는 AI 빌더가 아니므로 사업부/도메인별 RAG와 로컬 DB를 직접 운영하게 둘 수 없음
- **메커니즘**: Work Card → 사용 누적 → ③ 자산 데이터 등재
- **공개 범위**: 작성자 제안 + 검수자 확정. 개인 → 조직×도메인 권한 단위 → 도메인 종방향(부문) → 전사 표준으로 넓어질수록 추가 검수
- **승격 경로**: ③ 자산 → ② 표준 데이터는 드물게, ③ 자산 → Skill은 별도 경로
- **임원 라벨**: "People 조직 지식 자본화"

## 2.3 권한 — 필요한 사람에게 필요한 만큼만

| 데이터 | 권한 원칙 |
|---|---|
| DW 원천 데이터 | 담당 조직·도메인 권한이 있는 범위만 조회 |
| ① 운영 데이터 | 로그인·조직·권한 관리에 필요한 범위만 사용 |
| ② 표준 데이터 | 담당 도메인과 업무 단계에 맞는 기준만 RAG |
| ③ 자산 데이터 | 공개 범위와 검수 상태를 통과한 Knowledge만 retrieval |

보안과 축적은 권한 없이는 작동하지 않는다. 안전하게 막아도 오권한 retrieval이 나면 실패이고, 자산이 쌓여도 접근 권한이 없으면 AI가 활용하면 안 된다.

# §03 HR Data 및 권한 관리 체계

**Section Label**: Permission Model
**Headline**: 모든 데이터 접근은 조직×도메인 권한 단위에서 결정

## 3.1 X축 — 조직 (열람 범위 계층)

| 레벨 | 주체 | 열람 범위 |
|---|---|---|
| L0 Admin | AX&CI Lab | 전사 운영·표준 관리 |
| L1 부문(본사) | 부문 People팀 | 자기 부문 + 산하 모든 사업부 |
| L2 사업부 | 사업부 담당자 | 자기 사업부만 |

**사업부 8개**: VD, DA, MX, NW, 의료기기, SR, 생기연, CDO

## 3.2 Y축 — 도메인 (12개)

교육 / 노사 / 보상·근태 / 인력운영 / 임원조직 / 제도 / 채용 / 총무 / 해외인사 / DEI / 조직문화 / 집단지성

## 3.3 권한 단위 라벨링 (분류표 v0)

조직×도메인 조합이 Workbench의 기본 권한 단위다. 각 권한 단위에 4가지 라벨이 부착됨:

01. **데이터 접근 권한** — 어느 인사 DB 영역까지 열람 가능
02. **기본 민감도** — 부문이라고 모두 High 아님. 권한 단위별 개별 라벨링 필요
03. **차폐 정책** — 어느 항목을 어떻게 비식별화할지
04. **자산 승급 검수자** — Domain Experts 지정

숫자를 먼저 확정하지 않는다. 실제 운영 조직과 담당 도메인을 기준으로 권한 단위를 점진적으로 확정한다.

## 3.4 자산 공개 범위 (5단계)

| 범위 | 정의 | retrieval 가능자 |
|---|---|---|
| 개인 | 작성자 본인 | 본인만 |
| 권한 단위 한정 | 같은 조직×도메인 권한 단위 | 같은 권한 단위 담당자 |
| 도메인 종방향 | 같은 도메인, 조직 계층 따라 | 부문은 산하 사업부 권한 단위 자산 retrieval |
| 조직 횡방향 | 같은 조직, 모든 도메인 | 1차 출시 미지원 |
| 전사 표준 | 모든 권한 단위 | 누구나 |

## 3.5 권한 개념도

```
[사용자]
   ↓
[조직 역할 + 담당 도메인]
   ↓
[권한 단위 = 조직 × 도메인]
   ↓
┌──────────────────────┬────────────────────────────┐
│ DW                   │ 원천 HR 데이터 통합 저장소  │
│ HR AX Platform        │ ① 운영 ② 표준 ③ 자산 데이터 │
└──────────────────────┴────────────────────────────┘
   ↓
[필요한 만큼만 검색/RAG/Tool 실행]
   ↓
[분류 Agent → 라우팅 → 사내 LLM(GAUSS) 또는 사외 LLM]
```

# §04 데이터 저장 구조

**Section Label**: Data Architecture
**Headline**: 데이터는 DW와 HR AX Platform 두 군데에 저장된다

## 4.1 원천 시스템

| 시스템 | 담는 것 | 비고 |
|---|---|---|
| GHRP(S) | 급여, 근태, 평가 | GHRP(W) 토대 위에 구성 |
| GHRP(W) | 조직, 개인, 발령, 직무 | 마스터 |
| 기타 | 삼성U, 채용 Hub 등 | |

## 4.2 DW (Digital Warehouse)

- 위 3종 원천 통합 저장소
- 원천 HR 데이터를 모은 통
- HR AX Platform은 DW에서 필요한 범위만 가져와 자체 데이터 3종으로 적재·연계
- 원문 HR Data를 전사 공통 AI Platform에 업로드하지 않음

## 4.3 HR AX Platform 자체 적재 3종

### ① 운영 데이터 (Master)
시스템이 갖춰야 할 최소 데이터. 로그인·권한 관리를 위한 사용자 식별 정보, 조직·개인·발령 등 플랫폼 자체 기능 수행에 필수적인 마스터 데이터. HR AX Platform 내부 운영 데이터로 적재한다.

### ② 표준 데이터 (Standards)
부서·전사 차원에서 공식적으로 운영되는 정형·권위 자료. 인사 제도, 인사 기준(근태·평가·보상 등), 업무 프로세스(L4/L5/L6), 표준 양식으로 한정. AX&CI Lab이 운영하며 사람용 문서와 LLM RAG 양쪽에서 사용.

### ③ 자산 데이터 (Knowledge)
HR 담당자의 사용 흔적이 누적되어 형성되는 People 조직 자산. 사용 흔적 노하우, 판단 메모, 결정 narrative, 예외 케이스가 Tacit Candidate로 큐잉되고, Domain Experts 검수를 거쳐 자산 데이터에 등재. 조직×도메인 권한 단위와 공개 범위(개인 / 권한 단위 / 도메인 종방향 / 전사 표준)를 부착. 다수는 자산 데이터에 잔류하고, 일부만 표준 데이터 또는 Skill로 승격된다.

| # | 명칭 | 출처 | 주기 | 운영 |
|---|---|---|---|---|
| ① | 운영 데이터 (Master) | 플랫폼 내부 | 필요 시 동기화 | 시스템 자동 |
| ② | 표준 데이터 (Standards) | 직접 관리 | 잦은 수정 | AX&CI Lab |
| ③ | 자산 데이터 (Knowledge) | 사용 누적 | 실시간 | Domain Experts 검수 |

# §05 자산 승급 파이프라인

**Section Label**: Promotion Pipeline
**Headline**: 비정형 → 정형 → 인사제도/규정 승격은 모두 사람 검수를 거친다

## 5.1 단계

```
Step 1. Work Card 작성 / 종료
   └─ 사용 흔적이 자유 텍스트로 누적

Step 2. Tacit Candidate 자동 큐잉
   └─ 카드 종료 시 "결정적 판단 한 줄" 입력
   └─ 시스템이 결정 지점·반복 패턴 자동 감지

Step 3. Context Tag 부착
   └─ 조직×도메인 권한 단위, audience, processStep, bucketScope
   └─ 본문은 비정형 그대로
   └─ 작성자가 공개 범위 제안 (기본: 권한 단위 한정)

Step 4. Domain Experts 검수 (승인 / 반려 / 보류)
   └─ 승인: ③ 자산 데이터에 등재. 작성자·검수자 양쪽 실적
   └─ 반려: 작성자 피드백. 일정 기간 후 자동 폐기 또는 수정 후 재제출
   └─ 보류: 추가 정보 요청. 30일 미보강 시 자동 폐기

Step 5. ③ 자산 데이터에 등재
   └─ 공개 범위 (개인/권한 단위/도메인종방향/전사) 부착
   └─ 다른 카드에서 retrieval 가능

Step 6 (선택). Lab 4분기 결정
   └─ 새 L6: 규정/기준/업무 프로세스로 정형화 가능할 때 ② 표준 데이터로 승격
   └─ 새 Skill: LLM 운영 노하우로 추출 가능할 때 실행 레이어 Skill로 등록
   └─ 둘 다 / 둘 다 아님
```

## 5.2 정형 승급 — 비정형이 L6가 되는 흐름

- L4/L5/L6에 정의된 업무 = **정형 업무**
- 그 외 = **비정형 업무** (실제 People팀 일감의 다수)
- **반복 패턴이 신규 L6 또는 신규 Skill 후보로 자동 포착되고, Lab이 4분기 결정함**
- 승급된 L6는 Process Coaching AI 트리에 추가 → 다음 사람한테는 정형 업무로 매칭
- 승급된 Skill은 코드 리포지토리에 정의를 두고 DB 메타·임베딩으로 검색됨

## 5.3 부서원 작업물 차용 (Agent / Skill / Tool)

| 차용 항목 | 채택 | 노출 방식 |
|---|---|---|
| Agent / Skill / Tool 정의 | ⭕ | 실행 레이어 용어로 사용 |
| Skill 라이브러리 사용자 직접 선택 | ❌ | Work Card에 시스템 자동 부착 |
| 콜드 스타트 = 빈 채팅창 | ❌ | Work Card 보드가 콜드 스타트 |

**핵심**: "Skill 기반"이 아니라 "**Work Card에 Skill·Tool 자동 결합**"

# §06 우려와 정찰 항목

**Section Label**: Risks & Recon
**Headline**: 자산화는 자동으로 굴러가지 않는다 — 4가지 조건이 모두 갖춰져야

## 6.1 자산화 실패 모드

| 실패 모드 | 원인 | 대응 |
|---|---|---|
| 사용자가 결과만 받고 끝 | 노하우 추출 트리거 부재 | Card 종료 시 1줄 입력 강제, 결정 지점 자동 감지 |
| Tacit Candidate 큐 폭증 | 검수자 부담 | 자동 1차 분류, 유사 자산 머지, 검수자 적재 한도 |
| Domain Expert 검수 안 함 | 인센티브 없음 | People팀 차원 인정·평가 연계 |
| 자산이 retrieval에서 안 뜸 | 임베딩/태그 품질 | 한국어 임베딩 검증, 거짓 양성 모니터 |
| 자산이 떠도 사용자가 무시 | 신뢰 미형성 | 출시 초기 제한된 권한 단위에서 작동 검증 후 확대 |

## 6.2 정찰 항목 (P0)

01. **부문 People팀장 입장** — 강제 사용 선언이 가능한가
02. **분류표 v0 작성 가능 인력** — 조직×도메인 기본 등급(기밀/민감/일반) 라벨링 합의 누가
03. **Domain Experts 지명 권한** — 권한 단위별 검수자 누가 정하는가
04. **부서원 가설 통합 가능 여부** — Skill 정의 차용에 대한 합의

## 6.3 정찰 항목 (P1)

01. **DW 연계 정책** — 원천 시스템 데이터를 DW에서 어떤 범위와 주기로 받아올지
02. **사내 임베딩 모델 가용성** — 자산 retrieval용 (GAUSS 등)
03. **정보보호센터 협조 수준** — 매핑 사내 보관 정책 합의
04. **AX팀 게이트웨이 정책** — 사외 LLM 직접 접근 시 분류·로깅 방식

## 6.4 1차 출시 시점 정의

> "**제한된 조직×도메인 권한 단위에서, 등급 분류 + 비식별화 게이트 + Work Card + 검수형 자산화가 작동하는 v1**"

- 광범위 자산화는 무리
- 보안·축적·권한 흐름이 한 번에 검증되는 권한 단위부터 시작
- 숫자를 먼저 박지 않고 정책 합의와 운영 가능성을 보고 확대

# Appendix B — 청중별 한 페이지

## B.1 부문 People팀장·임원

> "**HR Data/Knowledge를 전사 공통 AI Platform에 올리지 않는 HR 특화 Platform**입니다.
> 민감 정보는 사외와 공통 AI 저장소에 나가지 않습니다.
> 부문 People팀 차원의 강제 사용 선언으로 진입을 잡고,
> 시간이 갈수록 People 조직의 지식 자본이 됩니다.
> 접근 권한이 있는 데이터와 Knowledge만 AI가 활용합니다."

## B.2 정보보호센터

> "**비식별화 게이트가 단일 통제점**입니다.
> 매핑 테이블은 사내에만 보관되며, 외부에는 차폐된 업무 맥락만 전달됩니다.
> 응답 수신 시 사내 재매핑으로 복원합니다.
> Application-side 감사 로그가 모든 호출을 기록합니다."

## B.3 AX팀

> "사내 LLM(GAUSS), 사외 LLM(Chat GPT/Gemini 등), 사내 AI Tool은 연계합니다.
> GAUSS는 LLM 자체 로그를 남기지 않는 운영이 가능하다는 점이 보안상 강점입니다.
> 다만 원문 HR Data/Knowledge, 권한 정책, 분류·차폐 정책은 HR AX가 통제합니다.
> 사외 LLM과는 Function Calling 패턴으로만 연결되며, Tool 실행은 모두 플랫폼 내부입니다.
> ※ 사외 LLM은 원문 HR Data/Knowledge를 직접 보지 않고, HR AX가 권한 확인·비식별화·감사 로그를 거친 차폐 컨텍스트만 받습니다."

## B.4 HR 담당자(사용자)

> "**Work Card에 일감 단위로 작업합니다.**
> 민감 정보 처리는 자동으로 안전하게.
> 지난 카드의 판단·노하우가 다음 카드에 자동으로 따라옵니다.
> 매번 처음부터 시작하지 않습니다."
