Harness Engineering

LLM이 HR 맥락 위에서 정확히 일하도록 — Context · 실행 · 외부 연계를 고려한 인프라 설계. HR 담당자는 업무 단위로 진입만, 인프라 자동 조립과 외부 시스템·AI 연계는 플랫폼이 처리.

3대 인프라

1. Context 공급

HR 맥락을 LLM에 공급

Process · 인사 규정 · 인사 기준 · 노하우(지식 데이터)

2. 실행체계 운영

Agent · Skill · Tool 3계층

LLM 처리 단위 + 절차 매뉴얼 + 결정적 코드

3. 외부 연계

시스템 · AI 서비스 허브

인사 시스템·DW · GAUSS · 사외 LLM · 사내 AI Tool

1. Context 공급 — HR 맥락을 LLM에 공급

요소설명
Process L3-L6 업무 프로세스 트리. Phase 1 산물 + 운영 중 신규 L6 추가.
인사 규정 HR 제도, 규정, 취업규칙 등.
인사 기준 평가·근태·보상 관련 기준, 표준 양식 등.
노하우(사례) 지식 데이터 = 검수 통과한 판단·결정·예외 케이스. 자산화 누적.

2. 실행체계 운영 — Agent · Skill · Tool 3계층

Agent

본질
LLM 처리 단위 (판단·결정의 주체)
Domain Agent

HR 도메인 업무를 직접 처리

검증된 업무부터 점진 구축. 없는 카드는 Skill·Tool만으로도 처리 가능.
예: 채용 직무적합성평가, 임원인사 근황 작성, 제도 모판 제작

System Agent

플랫폼 백엔드 운영 (사용자 비노출)

예: 데이터 민감도 분류 Agent, 차폐 검사 Agent, 자산 추출 Agent

Skill

본질
절차·기준 매뉴얼 (md)
설명
Lab이 정형화·등록한 작업 가이드. 사례가 충분히 쌓이면 신규 Skill로 승격.

지식 데이터(사례집)와 구분: Skill은 정형 매뉴얼, 지식 데이터는 개별 사례·판단 기록. 사례 → 매뉴얼 승격 관계.

Tool

본질
결정적 코드 동작
설명
정확한 동작 (조회·계산·연동). Agent가 있으면 Agent가 호출, 없으면 Skill 매뉴얼 지시에 따라 호출.

3. 외부 연계 — 시스템 · AI 서비스 허브

대상역할
인사 시스템 / Data Warehouse 원천 인사 데이터 통합 저장. HR AX는 DW에서 필요 범위만 인출.
사내 LLM (GAUSS) 기밀 데이터 처리 (로그 미저장). 사내 임베딩 + 분류 보강.
사외 LLM (Chat GPT, Gemini 등) 민감 데이터 (비식별화 후), 일반 데이터 처리.
사내 AI Tool (SPICA, AX Hub 등) 플랫폼 권한·차폐 정책 통과 시 연계.

두 진입 모드와 자동 조립 흐름

사용자 진입 | +-- Open Chat Mode | 사용자가 자유 입력 | → 업무 의도 분석 | → 권한 범위 안에서 L3-L6 후보 1~3개 추정 | → 민감도 분류 | → 필요 시 Guided Work로 전환 | +-- Guided Work Mode 로그인 권한·최근 업무·반복 패턴 기반 Work Card 제안 → L3-L6 후보 자동 부착 → Domain Agent 부착 (있을 때) → Skill + Tool + Context 자동 조립 → 민감도별 모델 라우팅 → 결정·판단 누적 → Skill/Agent/L6 후보

L6가 수천 개 수준이면 사용자가 L4/L5/L6를 직접 탐색하는 UX는 피로도가 높다. L6는 선택 UI가 아니라 플랫폼이 추정·확인·기록하는 메타데이터로 다루는 편이 맞다.

플랫폼 5 Layer 구조

Layer구성
L1. 사용자 진입 Work-space (조직×도메인 권한 단위) + Work Card
L2. 안전 게이트 데이터 민감도 분류 Agent · 비식별화 게이트 · Function Calling Broker · 플랫폼 자체 감사 로그
L3. 실행 Domain Agent · System Agent · Skill Registry · Tool 카탈로그
L4. 데이터 3종 ① 운영(Master) · ② 표준(Standards) · ③ 지식(Knowledge)
L5. 외부 연계 DW · 사내 LLM · 사외 LLM · 사내 AI Tool
