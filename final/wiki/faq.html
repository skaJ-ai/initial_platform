FAQ

A. 아키텍처 정합성

A1. 왜 HR AX는 별도 플랫폼인가?

전사 공통 AI Platform과 사외 LLM은 모델·기능으로 연계할 수 있지만, HR 데이터의 보안 등급, 조직×도메인 권한, 검수 책임, 지식 공개 범위는 HR AX가 통제해야 한다.

  • 보안: AI Tool 앞단의 안전 게이트. 기밀/민감 HR 데이터와 Knowledge의 원문 적재 금지, 분류·차폐·audit
  • 축적: Light user가 직접 RAG/DB를 관리하지 않아도 Work Card 사용 흐름에서 자산화
  • 권한: 조직×도메인 권한 단위별 DW·표준·지식 데이터 접근 통제
A2. 엔터프라이즈 LLM이면 안전한 것 아닌가? 계약된 모델인데 HR 데이터도 그냥 넣어도 되는가?

엔터프라이즈 계약은 중요한 출발점이지만, HR 민감데이터 사용을 자동으로 승인해 주는 장치는 아니다.

벤더가 입력 데이터를 모델 학습에 사용하지 않거나 보관 기간을 제한하더라도, 실제 위험은 입력 전후의 업무 흐름에서 발생한다. 예를 들어 프롬프트 원문이 애플리케이션·프록시·모니터링 로그에 남을 수 있고, 권한 범위를 넘는 데이터가 한 번에 입력될 수 있으며, 응답 결과에서 개인·평가·보상·조직 정보가 재식별될 수 있다.

따라서 People팀 관점에서는 "계약된 LLM을 쓴다"보다 "어떤 HR 데이터가, 어떤 권한으로, 어떤 맥락에서, 어떤 통제를 거쳐 LLM에 전달되는가"가 더 중요하다.

HR AX Platform은 상용 또는 사내 LLM을 단순 호출하는 화면이 아니라, HR 데이터의 민감도 분류, 권한 확인, 비식별화, 감사 로그, 검수 흐름을 LLM 앞뒤에 배치하는 통제 계층이다. 즉, LLM을 쓰기 위한 전제 조건을 플랫폼화하는 것이 목적이다.

A3. 사외 LLM과 Tool/Skill/Agent의 관계?

Function Calling 패턴. 사외 LLM은 Tool/Skill/Agent에 직접 접근 X. "Tool X를 인자 Y로 호출" 의도만 표현하고 실행은 플랫폼 내부에서.

A4. Skill md는 누가 작성하고 어디에 저장되는가?

초안: Domain Experts가 자동 추출 후보를 검토해 작성. 공식 등록: AX&CI Lab. 저장: 플랫폼 코드 리포지토리(정의) + 별도 DB(메타·임베딩).

B. 데이터 등급과 처리

B1. 데이터는 몇 단계로 분류되는가?

3단계 — 기밀 / 민감 / 일반.

등급처리모델
기밀사외 송출 금지사내 LLM (GAUSS, 로그 미저장)
민감비식별화 게이트 후 외부사외 LLM
일반그대로 외부사외 LLM
B2. 자동 분류는 어떻게 되나?

데이터 민감도 분류 Agent (System Agent)가 사내 LLM 전담으로 통합 처리:

  1. 1차: 권한 단위 기본 등급 룩업 (권한 분류표)
  2. 2차: 개인식별정보 패턴 대조 (사번/이름/평가 등)
  3. 3차: 사내 LLM(GAUSS) 한국어 맥락 보강

출력: {등급, 차폐 후보 항목, 신뢰도}

C. 자산화 운영

C1. Domain Experts는 누구인가?

부문 또는 사업부에서 선별된 담당자. 조직×도메인 권한 단위별 검수 책임. 주당 3-5시간 검수 (역할 정의·평가 연계).

C2. 자산 공개 범위 결정 주체?

작성자 제안 + 검수자 확정 + 단계적 권한 escalation. 디폴트 좁게(개인 또는 권한 단위 한정).

  • 좁히는 것은 단독 가능
  • 넓히는 것은 단계적: 권한 단위 → 도메인 종방향 → 전사 표준
C3. 검수 결과는 어떻게 분류되나?
결과처리
승인③ 지식 데이터에 등재
반려작성자 피드백 + 일정 후 자동 폐기 또는 수정 후 재제출
보류추가 정보 요청. 30일 미보강 시 자동 폐기

D. 진입과 강제

D1. 강제 사용은 어떻게 작동하나?

부문 People팀장 선언에 의존. 1차 출시는 선언 + 도구 매력으로 자발적 수렴 유도.

G. Overview 보충

G1. 단계적 추진 순서는 어떻게 이해하면 되나?

초기 Overview에 있던 단계 표현은 확정 일정이 아니라 설명용 추진 가설로 보는 편이 맞다. 현재는 다음 순서로 낮춰 표현한다.

  1. L3-L6 업무 트리와 AI 적용 후보를 정리한다.
  2. 반복성이 높고 검증 부담이 낮은 업무부터 Quick-Win으로 적용한다.
  3. 민감도 분류, 권한 확인, 비식별화, 감사 로그를 공통 안전 게이트로 묶는다.
  4. 반복 패턴이 누적된 업무를 Skill, Tool, Domain Agent 후보로 승격한다.

따라서 이 내용은 약속된 로드맵이라기보다 Lab과 다부서 논의에서 검증해야 할 실행 순서다.

G2. 한 장 아키텍처는 어떤 흐름을 말하나?

핵심은 사용자가 어떤 화면에서 시작하든 같은 통제 흐름을 탄다는 점이다.

사용자 요청
  → 로그인 조직·역할·도메인 확인
  → 업무 의도 및 L3-L6 후보 추정
  → 데이터 민감도 분류
  → 기밀 / 민감 / 일반 등급별 모델 라우팅
  → Tool 실행은 Function Calling Broker가 사내에서 통제
  → 감사 로그와 검수 후보를 남김
  → 승인된 판단·사례만 지식 데이터로 자산화

이 흐름 때문에 플랫폼은 단순 채팅창이 아니라 HR 데이터가 LLM으로 이동하기 전의 안전 관문이 된다.

G3. Open Chat Mode와 Guided Work Mode는 무엇이 다른가?
진입 방식적합한 상황플랫폼 처리
Open Chat Mode 탐색형 질문, 신규 이슈, 아직 Work Card로 정형화되지 않은 업무 입력 의도 분석 → 권한 확인 → L3-L6 후보 추정 → 필요 시 Guided Work로 전환
Guided Work Mode 반복 업무, 표준 프로세스, 검수·감사·자산화가 필요한 업무 권한·최근 업무·반복 패턴 기반 Work Card 제안 → Agent·Skill·Tool·Context 자동 조립

두 모드는 경쟁 관계가 아니다. 어느 쪽으로 시작해도 권한 확인, 민감도 분류, 모델 라우팅, 감사 로그, 검수형 자산화는 동일하게 적용된다.

H. 운영 현실 우려 (Realism)

부문장·임원·정보보호센터 협의에서 자주 나오는 회의론에 대한 답변.

H1. 분류 Agent 등으로 시스템 동작이 실제로 느려지지 않나?
단계추가 지연
데이터 민감도 분류 Agent~1-2초 (사내 LLM 짧은 분류 + PII 대조 + 룩업)
비식별화 게이트 (민감 등급일 때만)~수백 ms
의미 검색 (RAG)~100-500 ms
총 추가 지연1-3초 안팎

사외 LLM 본 호출 자체가 2-10초이므로 분류 지연은 전체의 10-30% 수준.

완화 방법: 동일 입력 캐싱, 권한 단위 사전 분류, 분류·게이트 병렬화, 작은 모델 사용.

H2. 노하우는 실제로 어떤 절차로 지식 데이터에 등재되나? 
1. 카드 종료 시
   - 사용자 "결정 한 줄" 입력
   - 시스템이 결정 지점·반복 패턴 자동 감지

2. 검수 후보 큐잉 (자유 텍스트 본문 + 메타)

3. Domain Experts 검수 (승인 / 반려 / 보류)

4. 승인 시:
   - ③ 지식 데이터 테이블에 등재
   - 공개 범위 부착

5. 의미 검색 저장소에 벡터화
   → 다음 카드 처리 시 retrieval로 자동 부착
H3. 자산화가 현실적으로 가능한가? (검수 부담 우려)

가장 큰 운영 우려. 다층 완화 메커니즘 적용.

  • Domain Experts 검수는 주당 3-5시간 수준. 역할 정의에 명시. People팀 평가에 연계.
  • 한 자산 검수 5-10분 가정 시 주 30-60건 처리 가능.
부담완화
검수자 시간 부족평가 반영으로 인센티브 보장
검수 큐 폭증자동 1차 분류·신뢰도 점수, 유사 자산 머지
1인 권한 단위 검수자 부담자가 검수 허용 (개인/권한 단위 한정만 기본)
보류 자산 무한 누적30일 미보강 자동 폐기
H4. 노하우 → 표준 또는 Skill 승격은 어떤 방식인가?

두 경로로 분리:

③ 지식 데이터 (사례 누적)
       │
       ├─ 경로 A: 규정/기준/업무 프로세스 형태로 정형 가능
       │           → ② 표준 데이터 신규 L6 노드 추가
       │           → 드뭄
       │
       └─ 경로 B: 절차·기준 매뉴얼로 정형 가능
                   → 신규 Skill md 등록
                   → 잦음

자동 포착은 동일 메커니즘. Lab이 4분기 결정: 새 L6 / 새 Skill / 둘 다 / 둘 다 아님.

H5. People팀 업무 다수가 비정형인데 Card 시스템이 무의미하지 않나?

정당한 우려. 비정형 비중 큰 것 사실. Card 시스템은 비정형도 1급 시민으로 처리.

핵심 원리: Agent가 꼭 붙지 않아도 카드는 동작. Skill + Tool 자동 부착만으로 LLM이 카드 처리 가능. Domain Agent는 검증된 업무부터 점진적으로 추가.

카드 종류처리 방식
정형 + Domain Agent 있음Domain Agent + Skill + Tool + Context
정형 + Domain Agent 없음Skill + Tool + Context만 LLM에 주입 — Agent 미부착
비정형도메인 일반 Skill·Context 부착 → 사용자 자유 입력

Domain Agent의 점진 확대: 1차 출시 시 검증된 업무에 한해 Domain Agent 우선 구축. 운영 중 반복 패턴 입증 업무에 신규 Agent 추가.

Open Chat → Guided Work 승격: 자유 입력의 결정·판단·반복 패턴 누적 → 신규 L6 / 신규 Skill / 신규 Domain Agent 후보 → Lab 결정으로 정형화. Card는 채팅을 대체하는 도구가 아니라 반복 업무를 안정적으로 처리하기 위한 진입 방식.
