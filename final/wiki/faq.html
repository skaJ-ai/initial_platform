FAQ

Lab·다부서·정보보호센터·부문장과의 대화에서 자주 나온 질문 누적. 카테고리별 정리.

A. 아키텍처 정합성

A1. 왜 HR AX는 별도 플랫폼인가?

전사 공통 AI Platform과 사외 LLM은 모델·기능으로 연계할 수 있지만, HR 데이터의 보안 등급, 조직×도메인 권한, 검수 책임, 지식 공개 범위는 HR AX가 통제해야 한다.

  • 보안: AI Tool 앞단의 안전 게이트. 기밀/민감 HR 데이터와 Knowledge의 원문 적재 금지, 분류·차폐·audit
  • 축적: Light user가 직접 RAG/DB를 관리하지 않아도 Work Card 사용 흐름에서 자산화
  • 권한: 조직×도메인 권한 단위별 DW·표준·지식 데이터 접근 통제
A2. 사외 LLM과 Tool/Skill/Agent의 관계?

Function Calling 패턴. 사외 LLM은 Tool/Skill/Agent에 직접 접근 X. "Tool X를 인자 Y로 호출" 의도만 표현하고 실행은 플랫폼 내부에서.

A3. Skill md는 누가 작성하고 어디에 저장되는가?

초안: Domain Experts가 자동 추출 후보를 검토해 작성. 공식 등록: AX&CI Lab. 저장: 플랫폼 코드 리포지토리(정의) + 별도 DB(메타·임베딩).

B. 데이터 등급과 처리

B1. 데이터는 몇 단계로 분류되는가?

3단계 — 기밀 / 민감 / 일반.

등급처리모델
기밀사외 송출 금지사내 LLM (GAUSS, 로그 미저장)
민감비식별화 게이트 후 외부사외 LLM
일반그대로 외부사외 LLM
B2. 자동 분류는 어떻게 되나?

데이터 민감도 분류 Agent (System Agent)가 사내 LLM 전담으로 통합 처리:

  1. 1차: 권한 단위 기본 등급 룩업 (분류표 v0)
  2. 2차: 개인식별정보 패턴 대조 (사번/이름/평가 등)
  3. 3차: 사내 LLM(GAUSS) 한국어 맥락 보강

출력: {등급, 차폐 후보 항목, 신뢰도}

C. 자산화 운영

C1. Domain Experts는 누구인가?

부문 또는 사업부에서 선별된 담당자. 조직×도메인 권한 단위별 검수 책임. 주당 3-5시간 검수 (역할 정의·평가 연계).

C2. 자산 공개 범위 결정 주체?

작성자 제안 + 검수자 확정 + 단계적 권한 escalation. 디폴트 좁게(개인 또는 권한 단위 한정).

  • 좁히는 것은 단독 가능
  • 넓히는 것은 단계적: 권한 단위 → 도메인 종방향 → 전사 표준
C3. 검수 결과는 어떻게 분류되나?
결과처리
승인③ 지식 데이터에 등재
반려작성자 피드백 + 일정 후 자동 폐기 또는 수정 후 재제출
보류추가 정보 요청. 30일 미보강 시 자동 폐기

D. 진입과 강제

D1. 강제 사용은 어떻게 작동하나?

부문 People팀장 선언에 의존. 1차 출시는 선언 + 도구 매력으로 자발적 수렴 유도.

E. 운영 현실

E1. Tool 추가 요청 흐름? 
HR 담당자 → Tool 요청 (양식)
   → AX&CI Lab 검토 (보안·중복·우선순위)
   → AX&CI Lab 구현
   → Tool 카탈로그 등록
   → Agent에서 호출 가능

F. 네이밍 결정

F1. 데이터 3종 명칭 (확정)
  • ① 운영 데이터 (Master)
  • ② 표준 데이터 (Standards)
  • ③ 지식 데이터 (Knowledge)
F2. 호칭·트랙 표현 규약
  • "OO 프로" 사용. ❌ "대리/과장" 금지
  • "부문 People팀" 사용. ❌ "People팀 본부" 금지
  • "민감 데이터 여부" 표현 사용 (정치/코드 트랙 분리 X)

H. 운영 현실 우려 (Realism)

부문장·임원·정보보호센터 협의에서 자주 나오는 회의론에 대한 답변.

H1. 분류 Agent 등으로 시스템 동작이 실제로 느려지지 않나?
단계추가 지연
데이터 민감도 분류 Agent~1-2초 (사내 LLM 짧은 분류 + PII 대조 + 룩업)
비식별화 게이트 (민감 등급일 때만)~수백 ms
의미 검색 (RAG)~100-500 ms
총 추가 지연1-3초 안팎

사외 LLM 본 호출 자체가 2-10초이므로 분류 지연은 전체의 10-30% 수준.

완화 방법: 동일 입력 캐싱, 권한 단위 사전 분류, 분류·게이트 병렬화, 작은 모델 사용.

H2. 노하우는 실제로 어떤 절차로 지식 데이터에 등재되나? 
1. 카드 종료 시
   - 사용자 "결정 한 줄" 입력
   - 시스템이 결정 지점·반복 패턴 자동 감지

2. 검수 후보 큐잉 (자유 텍스트 본문 + 메타)

3. Domain Experts 검수 (승인 / 반려 / 보류)

4. 승인 시:
   - ③ 지식 데이터 테이블에 등재
   - 공개 범위 부착

5. 의미 검색 저장소에 벡터화
   → 다음 카드 처리 시 retrieval로 자동 부착
H3. 자산화가 현실적으로 가능한가? (검수 부담 우려)

가장 큰 운영 우려. 다층 완화 메커니즘 적용.

  • Domain Experts 검수는 주당 3-5시간 수준. 역할 정의에 명시. People팀 평가에 연계.
  • 한 자산 검수 5-10분 가정 시 주 30-60건 처리 가능.
부담완화
검수자 시간 부족평가 반영으로 인센티브 보장
검수 큐 폭증자동 1차 분류·신뢰도 점수, 유사 자산 머지
1인 권한 단위 검수자 부담자가 검수 허용 (개인/권한 단위 한정만 기본)
보류 자산 무한 누적30일 미보강 자동 폐기
H4. 노하우 → 표준 또는 Skill 승격은 어떤 방식인가?

두 경로로 분리:

③ 지식 데이터 (사례 누적)
       │
       ├─ 경로 A: 규정/기준/업무 프로세스 형태로 정형 가능
       │           → ② 표준 데이터 신규 L6 노드 추가
       │           → 드뭄
       │
       └─ 경로 B: 절차·기준 매뉴얼로 정형 가능
                   → 신규 Skill md 등록
                   → 잦음

자동 포착은 동일 메커니즘. Lab이 4분기 결정: 새 L6 / 새 Skill / 둘 다 / 둘 다 아님.

H5. People팀 업무 다수가 비정형인데 Card 시스템이 무의미하지 않나?

정당한 우려. 비정형 비중 큰 것 사실. Card 시스템은 비정형도 1급 시민으로 처리.

핵심 원리: Agent가 꼭 붙지 않아도 카드는 동작. Skill + Tool 자동 부착만으로 LLM이 카드 처리 가능. Domain Agent는 검증된 업무부터 점진적으로 추가.

카드 종류처리 방식
정형 + Domain Agent 있음Domain Agent + Skill + Tool + Context
정형 + Domain Agent 없음Skill + Tool + Context만 LLM에 주입 — Agent 미부착
비정형도메인 일반 Skill·Context 부착 → 사용자 자유 입력

Domain Agent의 점진 확대: 1차 출시 시 검증된 업무에 한해 Domain Agent 우선 구축. 운영 중 반복 패턴 입증 업무에 신규 Agent 추가.

비정형 → 정형 승격: 비정형 카드의 결정·판단·반복 패턴 누적 → 신규 L6 / 신규 Skill / 신규 Domain Agent 후보 → Lab 결정으로 정형화. Card는 비정형을 정형으로 전환하는 도구.
