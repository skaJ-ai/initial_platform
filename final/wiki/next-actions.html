정찰 항목 + 의사결정 요청 + 1차 출시까지의 주요 트랙.

정찰 항목 (P0 — 가장 결정적)

항목왜 필요한가카운터파트
부문 People팀장 입장 강제 사용 선언이 가능한가 People팀장
권한 분류표 작성 가능 인력 조직×도메인 기본 등급(기밀/민감/일반) 라벨링 합의 누가 People팀 + 정보보호센터
Domain Experts 지명 권한 권한 단위별 검수자 누가 정하는가 People팀장
부서원 가설 통합 Skill 정의·작성물 차용에 대한 합의 부서 내

정찰 항목 (P1)

항목설명
DW 연계 정책원천 시스템 데이터를 DW에서 어떤 범위·주기로 받아올지
사내 임베딩 모델 가용성지식 retrieval용 (GAUSS 등)
정보보호센터 협조 수준비식별화 매핑 사내 보관 정책 합의
AX팀 게이트웨이 정책사외 LLM 직접 접근 시 분류·로깅 방식

의사결정 요청 (Lab → 부문장)

  1. 강제 정책 발효 승인 — HR 데이터 처리 시 HR AX 통과 필수 선언
  2. 정보보호센터 카운터파트 지정 — 권한 분류표 합의 시작
  3. Domain Experts 지명 권한 위임 — 권한 단위별 검수자 운영
  4. AX팀과 영역 분리 합의 — AI Tool 인프라(AX팀) vs HR 도메인 정책(People팀)

1차 출시까지 주요 트랙 (Phase 3)

단계갖춰야 할 것핵심 백로그
1 보안 기반 권한 분류표, 데이터 민감도 분류 Agent, 플랫폼 자체 감사 로그
2 안전 게이트 라우팅 정책, 비식별화 게이트, Function Calling Broker
3 권한 체계 조직×도메인 권한 단위, DW/표준/지식 데이터 접근 제어
4 자산화 파이프라인 Work Card, 검수 후보 큐, Review Queue, 지식 신뢰 단계
5 운영화 Skill 라이브러리, Tool 카탈로그, 분류표 갱신 루프, 운영 모니터링

1차 출시 정의

제한된 조직×도메인 권한 단위에서, 등급 분류 + 비식별화 게이트 + Open Chat/Guided Work + 검수형 자산화가 작동하는 1차 운영안. 숫자를 먼저 박지 않고 정책 합의와 운영 가능성을 보고 확대.

리스크 핫스팟

리스크대응
정치 트랙 지연 → 강제 사용 못 만듦 강제형 진입 대신 Open Chat과 Guided Work를 병행하는 방향으로 조정
AX팀이 자체 게이트웨이에 HR 정책까지 넣겠다 "도메인 정책은 People팀 책임" 영역 분리 재확인. Policy API로 위임 형태 협상
GAUSS 성능이 떨어져 민감 등급 대부분이 비식별화 게이트+사외 LLM으로 감 정책 자산 가치 자체는 유효. GAUSS 운영팀과 동맹 강화
Domain Experts 검수 안 함 (인센티브 없음) People팀 평가에 검수 행위 반영. 검수 적재 한도 운영
비정형 카드에서 Agent 매칭 실패 다수 Skill·Tool만으로도 동작 가능한 구조. Domain Agent는 점진 추가
