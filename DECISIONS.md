# Decisions — 가설 진화 + 대안 평가

> v1 → v2 → v2.1 → v3 진화 과정과 폐기/흡수 이유.
> 부서원 대안 가설(Skill 기반 HR Platform) 평가.

---

## 1. 가설 진화 한눈에

| 버전 | 한 줄 | 상태 |
|---|---|---|
| v1 | 데이터 자산 OS — L4/L5/L6를 본체로 두고 사외 LLM은 BYOM 슬롯 | **폐기** |
| v2 | HR 운영 레이어 — 데이터 위 권한·승인·감사·워크플로우 | **흡수** (v3의 일부) |
| v2.1 | 민감도 기반 LLM 라우팅 — GAUSS/사외 LLM 자동 분기 | **흡수** (v3의 Routing) |
| **v3** | **Mandate + Routing + Accumulation** | **채택** |

## 2. 각 버전의 채택/폐기 이유

### v1 — 데이터 자산 OS (폐기)

**가설**: L4/L5/L6 분해 데이터를 본체로 두고, 사외 LLM은 BYOM 슬롯으로 위에 얹는다.

**근거 (당시)**:
- Process Coaching AI가 만든 L4/L5/L6 분해 = 컴페티터가 못 가짐
- "회사 맥락 누적" (AXIOM positioning.md 테제)

**왜 폐기됐나**:
- "사외 LLM은 항상 사외 클라우드"라는 가정이 단순했다
- 실제로는 7월부터 사내망 안에서 사외 LLM/사내 AI Tool 활용 환경이 넓어짐
- 같은 사내망 동거인이 같은 데이터에 접근 가능 → "우리만 가졌다" 차별 무너짐

### v2 — HR 운영 레이어 (흡수)

**가설**: 차별은 데이터가 아니라 데이터 위 운영 레이어 — 권한·승인·감사·워크플로우.

**채택**:
- 데이터 격리 moat은 사라졌지만, 운영 규약은 People팀이 정의해야 하는 영역
- 사외 LLM이 같은 데이터에 접근하더라도 도메인 규약은 못 만듦

**보강 이유**:
- 너무 추상적, 구체적 차별 메커니즘 부족
- 사외 LLM/AI Tool 자체 거버넌스(SSO/Audit/Retention)가 같이 들어와서 "권한·감사" 단독으로 약함

### v2.1 — 민감도 기반 LLM 라우팅 (흡수)

**가설**: 사내 LLM(GAUSS)와 사외 LLM 공존 환경에서 HR 도메인 분류기로 라우팅.

**채택**:
- GAUSS "LLM 자체 로그 미저장" + 사외 LLM 누수 채널을 인식하면, 라우팅 자체가 도메인 정책 자리
- AX팀(인프라) vs People팀(정책) 영역 분리 깔끔

**보강 이유**:
- 라우팅만으론 사용자가 안 들어옴 (콜드 스타트)
- 자산화 메커니즘 누락
- "왜 강제냐"의 정당성 자체는 약함

### v3 — Mandate + Routing + Accumulation (채택)

**가설**: HR 데이터 처리 시 HR AX 안전 게이트 진입 + 자동 라우팅 + 자산 누적 + 조직×도메인 권한 통제의 합성.

**채택 근거**:
- 셋 중 하나라도 빠지면 무너지는데 셋이 합쳐지면 선순환
- 사외 LLM이 절대 카피 못 하는 메커니즘 셋의 결합
- 시간이 갈수록 강해지는 유일한 구조

## 3. v3가 단단한 이유

| 이유 | 설명 |
|---|---|
| 셋의 결합이 선순환 | 강제로 사용자 진입 → 라우팅으로 안전+성능 → 누적으로 다음 사용 좋아짐 → 자발적 사용 ↑ |
| 사외 LLM이 못 카피 | 원문 HR Data/Knowledge 독립 통제, 도메인 분류, 조직×도메인 권한, 누적은 People팀 맥락 |
| 정치적 동맹 가능 | 정보보호센터, 보안, 컴플라이언스가 자연 동맹 |
| 시간이 우리 편 | 자산 누적으로 시간 갈수록 moat 강화 |

## 4. 부서원 대안 평가 — Skill 기반 HR Platform

### 4.1 대안 요약

부서원이 제안한 가설:
- 콜드 스타트 = 채팅창. 위에 Skill 라이브러리.
- 사용자 직접 선택 또는 LLM 자동 선택
- 사외 LLM 기반 자유 채팅
- Process Coaching Tool 자산을 Agent로 활용
- Skill 연계 시나리오 (정보 조회 + 번역 + 계산 등)

Agent / Skill / Tool 3계층 정의도 이 부서원 작업이며, 정의 자체는 우수하다.

### 4.2 강점 (인정)

1. UX 직관성 압도적
2. 빌드 속도 빠름 (산업 표준 패턴)
3. 시나리오 1·3의 사용자 가치 진짜
4. Process Coaching 자산을 살림
5. Agent/Skill/Tool 모듈화 사고 단단

### 4.3 약점

#### (1) 사외 LLM과 정면 대결 — 가장 큰 우려

콜드 스타트 = 챗 + Skill 라이브러리 + 사외 LLM 자유 채팅 + 도구 호출 = Gems, Custom GPTs 같은 패턴.

People팀장이 던질 질문: *"이거 사외 LLM에 도구 등록하면 똑같이 안 돼?"* → 답하기 매우 어려움. 차별 인식 무너짐.

#### (2) 거버넌스/Mandate 빠짐

"홍길동 임직원 정보 찾아서 영어 번역" 시나리오가 사외 LLM 자유 챗에서 돌면 임직원 개인정보가 사외 LLM 컨텍스트에 그대로 올라감. 보안 게이트 부재.

(부서원 가설에도 거버넌스 적용 가능하지만 — 그러면 v3와 같아짐)

#### (3) 자산 누적 메커니즘 없음 — 가장 결정적

매번 콜드 스타트. 사용 흔적이 회사 자산으로 누적되는 구조 부재. 1년 써도 첫날과 같은 도구.

[AXIOM/Agentwork.md](../AXIOM/Agentwork.md)에서 강조한 "moat는 자산 승격 구조에서 생긴다"는 핵심이 빠진 가설.

#### (4) Mandate 정당성 못 만듦

"Skill이 모여 있어서 편함"은 강제 정당성 안 됨. 결국 "쓰고 싶은 사람만 쓰는 도구" → 사외 LLM이 더 좋아 보이면 이탈.

#### (5) Skill을 사용자에게 직접 노출 — UX 함정

직접 선택 → 메뉴 헤맴. LLM 자동 선택 → 사실상 챗과 구별 안 됨. 둘 다 단단하지 않음.

#### (6) HR Data 및 권한 관리 체계 부재

DW와 Platform 자체 적재 3종, 조직×도메인 권한 단위, 검수 큐, 분류 정책, 자산 승급 흐름이 그림에 없음.

#### (7) "또 하나의 챗 UI" 함정

7월 후 차별 인식 못 만들 가능성 매우 높음.

### 4.4 두 가설의 결정적 비교

| | 부서원 답 | v3 답 |
|---|---|---|
| "왜 이 플랫폼인가?" | "Skill이 모여 있어서" | "안전 게이트 + 축적 + 권한을 HR AX가 통제" |
| "사외 LLM과 뭐 달라요?" | 답하기 어려움 | 답 즉시 가능 |

이 한 줄이 둘의 운명을 가른다.

### 4.5 결론

**v3 채택. 부서원 가설은 단독으로 가지 않음.**

다만 부서원의 좋은 부분(Agent/Skill/Tool 정의, 시나리오 1·3의 가치)은 **v3의 실행 레이어로 그대로 사용**한다. 부서원의 모듈화 작업이 무가치한 것이 아니라, **그 위에 보안·축적·권한이 얹혀야 회사에서 살아남는다**는 판단.

## 6. v3 전환 시 따라오는 조정 사항

### 6.1 현재 빌드 중인 "스킬+툴콜링" 플랫폼

- 폐기 아님 — Agent/Skill/Tool 실행 레이어로 살림
- 단 위에 안전 게이트·축적·권한을 얹는 작업이 P0가 됨
- 콜드 스타트 UI는 빈 채팅창 → Work Card 기반으로 재설계

### 6.2 AXIOM

- 본체에서 사이드로 위치 재고정
- Accumulation 데이터 모델은 차용 ([ASSETS.md](./ASSETS.md))
- "공통 Agent" 컨셉은 본체 출시 후 검토

### 6.3 Process Coaching AI

- 단독 도구에서 "본체 플랫폼의 프로세스 자산 입력기"로 재위치
- L4/L5/L6 JSON → 본체 work_cards/process_assets 테이블 import 파이프라인 필요

## 8. HR 데이터 특화 Platform 결정

**결정**: HR AX는 전사 공통 AI Platform 위 플러그인이 아니라 HR 데이터 특화 Platform으로 가져간다.

**근거**:
- 원문 HR Data/Knowledge는 전사 공통 AI Platform에 올리지 않는다.
- HR 담당자는 Light user이므로 사업부/도메인별 RAG와 로컬 DB를 직접 운영하게 둘 수 없다.
- DW, 표준 데이터, 자산 데이터는 조직×도메인 권한 단위로 필요한 만큼만 접근돼야 한다.
- 사내 LLM(GAUSS), 사외 LLM(Chat GPT/Gemini 등), 사내 AI Tool은 연계 대상이지만, 권한·분류·차폐·audit은 HR AX가 통제한다.
- GAUSS는 LLM 자체 로그를 남기지 않는 운영이 가능하다는 점이 보안상 강점이다.
- 사외 LLM은 원문 HR Data/Knowledge를 직접 보지 않고, HR AX가 권한 확인·비식별화·감사 로그를 거친 차폐 컨텍스트만 받는다.
