## 1. AX팀 / AI Tool 도입 범위

### Q1.1 들어오는 게 모델인가, 제품인가
- "모델 API"만 들어오나, "워크스페이스 제품"까지 들어오나?
- 워크스페이스 제품이면 Custom GPT / Gems / Skills 등 사용자 자체 도구 등록 가능?

### Q1.2 거버넌스 동반 여부
- 권한 통제(RBAC) 같이 들어오나?
- Audit log는 어디 저장되고 누가 조회 가능?
- 데이터 retention 정책 (0 retention 협상 가능?)

### Q1.3 부서별 정책 세팅 가능 여부 (가장 결정적)
- **부서별로 다른 사용 정책 세팅 가능?** ("HR 데이터 다루는 부서만 다른 룰" 가능?)
- 가능 → People팀 영역 좁아짐, 후보 2(워크플로우)·4(시스템 연동) 강화 필요
- 불가 → People팀 영역 살아남음, v3 전체 단단

### Q1.4 AX팀 자체 AI Gateway 깔 계획?
- LiteLLM / Portkey / Kong AI Gateway / 자체 빌드?
- Gateway 안에 도메인 정책까지 포함시킬 계획?
- → "도메인 정책은 People팀 책임" 영역 분리 협상 필요

## 2. 사내 LLM (GAUSS) 구조

### Q2.1 운영 주체
- GAUSS를 어느 부서가 운영?
- 우리가 직접 호출 가능한지, AX팀 게이트웨이 통과해야 하는지?

### Q2.2 보안 등급별 분리
- GAUSS가 단일 환경인가, 보안 등급별 분리 환경인가?
- (예: Tier 1 일반업무용 / Tier 2 인사데이터용 / Tier 3 극민감용)
- 분리돼 있다면 → 라우팅이 1단 아니라 2단 (민감도 → GAUSS 등급 선택)

### Q2.3 사외 LLM 범위
- 사외 LLM 범위는 Chat GPT / Gemini 등으로 어디까지 볼 것인가?
- 각 사외 LLM별 로그/retention/감사 정책은 동일한가?
- 분류표 v0에서 같은 사외 LLM 그룹으로 묶을 수 있는가?

### Q2.4 GAUSS 성능 한계
- 어떤 종류의 작업까지 GAUSS로 가능?
- 한국어 / 영어 / 표 분석 / 긴 컨텍스트 / 멀티모달 각각의 성능
- → 별표 영역(중간 민감 + 복잡)에서 GAUSS vs 마스킹+사외 LLM 선택 기준 결정

### Q2.5 분류 Agent용 사내 모델
- 데이터 민감도 분류 Agent에서 GAUSS를 어떤 방식으로 호출할 수 있는가?
- PII 패턴 매칭 후 한국어 맥락 보강 용도에 충분한 정확도와 latency가 나오는가?
- 분류 Agent 호출 로그와 audit 저장 정책은 누가 승인하는가?
